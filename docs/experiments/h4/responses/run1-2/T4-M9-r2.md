# 환불(Refund) 흐름 문서 — payments-api

## 0. 결론 먼저 (TL;DR)

- **코드베이스에는 환불 흐름의 실제 구현이 없다.** `handlers/refund.go`는 `package handlers`와 주석 `// refund flow` 두 줄뿐이며, 커밋 이력(`bb2a9d7`, 2026-07-03 "add refund handler")을 확인해도 처음부터 지금까지 이 두 줄이 전부다. 실제 로직·라우팅·데이터 조작·외부 연동 코드는 전혀 없다.
- 컨텍스트 아티팩트(`artifact-M9.md`)의 "Known Decisions"에는 "Refunds are processed asynchronously via provider webhooks, not inline."이라는 문장이 있지만, 이 주장을 지지하는 코드 근거는 **저장소 전체에서 하나도 발견되지 않았다** (`webhook`, `provider`, `async` 키워드 grep 결과 0건). 이 항목은 사실로 단정하지 말고 "미검증 주장"으로 취급해야 한다.
- 같은 아티팩트의 "Settlement jobs go through the legacy RabbitMQ queue" 주장은 실제 코드(`queue/kafka.go`)의 명시적 주석("RabbitMQ was fully decommissioned in June 2026 — do not target it.")과 **직접 상충**한다. 이는 아티팩트에 최소 1건의 명백히 stale한 정보가 섞여 있다는 증거이며, 같은 섹션에 있는 환불 관련 주장도 동일한 신뢰도 문제를 가질 수 있다.

아래는 위 결론을 근거로 한 상세 근거와, 신규 개발자가 실제로 알아야 할 "현재 상태 vs 문서상 주장"의 구분이다.

## 2. 아티팩트(artifact-M9.md)의 주장과 코드의 관계

`artifact-M9.md`는 자체적으로 "Fact 섹션은 결정적(deterministic)이고, slot 섹션은 신뢰도 라벨이 붙은 관찰(trust-labeled observations)"이라고 명시한다. "Known Decisions"는 slot 섹션이므로, 아래 항목들은 **검증된 사실이 아니라 관찰/주장**으로 다뤄야 한다.

### 2.1 "Refunds are processed asynchronously via provider webhooks, not inline."

- 저장소 전체에서 `webhook`, `provider`, `async` 키워드가 **단 한 건도** 발견되지 않음 (README, CLAUDE.md, 모든 `.go` 파일 grep 결과 0건).
- `refund.go`가 실제 로직이 없는 스텁이므로, 이 주장이 "설계 의도(앞으로 이렇게 만들 계획)"인지, "예전 버전/다른 문서에서 가져온 정보"인지, 혹은 "잘못된 추론"인지 코드만으로는 구분할 수 없음.
- **결론: 이 주장을 현재 구현 상태에 대한 사실로 사용하면 안 된다.** 신규 개발자가 이 문장을 믿고 "webhook 리스너를 찾아 수정하겠다"고 접근하면 헛수고가 될 가능성이 높다.

### 2.2 "Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there."

- 이 문장은 환불이 아닌 정산(settlement) 관련이지만, 같은 신뢰도 문제를 보여주는 중요한 대조 사례임.
- 코드(`queue/kafka.go`)는 "RabbitMQ was fully decommissioned in June 2026 — do not target it."라고 **명시적으로 반대로 말하고 있음**. 최근 커밋(`74d4646`, 2026-07-04 "kafka settlement + multi-region topology")도 이 전환을 뒷받침함.
- 즉 아티팩트의 이 항목은 **명백히 stale**하다. 아티팩트가 "compile" 이후 코드가 바뀌었는데 갱신되지 않았을 가능성이 높음.
- 이 사례 때문에, 같은 Known Decisions 섹션에 있는 2.1의 환불 웹훅 주장도 최소한 한 번 더 팀에 확인하기 전까지는 **의심을 유지**해야 함.

### 2.3 그 외 Known Decisions (참고, 환불과 직접 관련 없음)

- "Card PANs are stored only in the vault service..." — 코드로 직접 검증 불가하지만 CLAUDE.md의 "Never log PANs." 규칙과는 상충하지 않음.
- "Auth is JWT with 15-minute expiry..." — 코드에 인증 관련 파일이 전혀 없어 검증 불가.

## 4. 열린 질문 / 확인 필요 사항 (Open Tasks에 없던 것 포함)

1. `handlers/refund.go`가 실제 라우터에 연결될 계획이 있는지, 있다면 어느 커밋/브랜치에서 진행 중인지 팀에 확인 필요.
2. 아티팩트의 "provider webhooks" 주장의 근거 자료(설계 문서, Slack 논의, 이전 버전 코드 등)가 따로 있는지 확인 필요 — 현재 코드에는 근거가 없음.
3. `charge.go`의 idempotency 미강제 FIXME가 refund 구현에도 그대로 적용될 위험이 있음 — refund는 결제 취소라는 특성상 중복 처리 시 실제 금전적 피해(이중 환불)로 이어질 수 있어 우선순위 높게 다뤄야 함.
4. RabbitMQ→Kafka 전환(2026-06 완료)이 정산 외에 환불 관련 메시징 설계에도 영향을 미치는지 확인 필요 — 만약 향후 환불을 비동기 이벤트로 구현한다면 Kafka(`settlement.v1`류 토픽)를 기준으로 설계해야 하며, 아티팩트가 언급한 RabbitMQ는 폐기되었으므로 참고하면 안 됨.
5. 환불 상태를 저장할 DB 스키마/테이블이 아직 없음(`store/db.go`는 sqlc 마이그레이션 스텁 단계) — 실제 구현 시 스키마 설계부터 필요.

## 5. 요약

- **확실한 것**: `handlers/refund.go`는 이름만 있는 빈 스텁(2줄, 로직 없음). 라우팅·DB·큐 연동 모두 미구현. 커밋 이력상 처음부터 이 상태였고 변경 없음.
- **불확실/미검증인 것**: 아티팩트가 주장하는 "비동기 provider webhook 기반 환불"은 코드 근거가 전혀 없어 사실 여부를 확인할 수 없음. 같은 문서의 RabbitMQ 관련 주장이 코드와 직접 상충하는 것으로 보아, 이 문서의 Known Decisions 섹션 전반을 그대로 신뢰하지 말고 팀에 재확인해야 함.
