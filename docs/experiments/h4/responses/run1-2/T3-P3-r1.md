# 토큰 만료 버그, 어디부터 봐야 하나

## 결론부터: 이 저장소 안에는 볼 곳이 없다 (중요한 발견)

`artifact-P3.md`의 "Known Decisions"에는 다음 항목이 **[verified — human:kim]**로 기록되어 있다.

> Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh.

하지만 실제 코드(`/Users/[user]/Developer/30_lab/EXPER/h4-kit/repo`)를 전부 열어봐도 이 내용을 구현한 코드가 **하나도 없다**:

- `main.go` — `func main() {}` 뿐이고 라우터/미들웨어 등록 자체가 없음 (`// TODO: wire settlement cron`만 있음)
- `handlers/charge.go`, `handlers/refund.go` — 둘 다 실제 로직 없이 주석 한 줄짜리 스텁 (`// FIXME: idempotency keys not enforced`, `// refund flow`)
- `store/db.go`, `queue/kafka.go` — 각각 sqlc 스텁, Kafka 발행 스텁뿐
- `go.mod` — `module payments`, `go 1.23` 외 의존성 선언이 전혀 없음. 즉 JWT 라이브러리(`golang-jwt` 등)조차 임포트되어 있지 않다.
- 파일 시스템 어디에도 `auth`, `jwt`, `token`, `middleware` 같은 이름의 파일/패키지가 없음 (grep 결과 없음)

즉 이 레포는 "settlement/kafka 골격 + 스텁 핸들러" 수준의 초기 스켈레톤이고, 인증/JWT 관련 구현은 **커밋 이력에도 등장한 적이 없다** (`f4ba12b` initial skeleton → `bb2a9d7` add refund handler(스텁) → `54be496` stub sqlc → `74d4646` kafka+multi-region). Known Decisions는 아티팩트 스킴상 "trust-labeled observation" 슬롯이지 코드에서 검증된 Fact 섹션이 아니다 — 사람이 검증했다고 표시돼 있어도, 그 검증 대상이 "이 저장소의 코드"라는 보장은 없다(다른 문서/스펙/위키를 보고 적었을 수 있음).

## 그래서 실제로 해야 할 일 (우선순위 순)

1. **"토큰"이 무엇을 가리키는지부터 확인** — 버그 리포트 원문을 봐야 한다. 후보가 최소 3개 갈린다:
   - (a) 사용자/서비스 인증용 JWT 세션 토큰 (Known Decisions에 있는 15분 만료 + `/auth/refresh`)
   - (b) `handlers/charge.go`의 FIXME에 있는 "idempotency key" (토큰은 아니지만 혼동되기 쉬움, 만료 개념과는 다름)
   - (c) 결제 대행사(PSP)로 나가는 외부 API 토큰/웹훅 서명 — `refund.go`가 "webhook 기반 비동기 처리"(⚠ conflict 표시됨)와 관련될 수 있음
   
   이 중 (a)라면 아래 2번으로, (b)/(c)라면 이 저장소만으로는 답이 안 나온다.

2. **인증 코드가 실제로 어느 저장소/서비스에 있는지 찾기.** payments-api 자체에는 JWT 검증/발급 코드가 없으므로 다음 중 하나일 가능성이 높다:
   - 별도의 auth 서비스나 API 게이트웨이가 앞단에서 JWT를 검증하고, payments-api는 이미 검증된 요청만 받는 구조 (그렇다면 버그는 이 레포가 아니라 게이트웨이/auth 서비스 쪽)
   - 아직 이 레포에는 구현되지 않았고 다른 스프린트/브랜치에서 작업 중 (README/CLAUDE.md 어디에도 인증 관련 언급 없음)
   - 문서(Known Decisions)만 있고 실제 구현은 계획 단계 (스켈레톤 상태의 다른 파일들과 패턴이 일치)
   
   → kim(해당 decision을 verified 표시한 사람)에게 "이 JWT 로직이 실제로 어느 코드에 있는지" 먼저 확인하는 게 가장 빠르다.

3. **레포 안에서 시작점을 굳이 골라야 한다면** `main.go`가 유일하게 진입점 역할을 하는 파일이므로, 라우터/미들웨어가 나중에 추가된다면 여기서 등록될 것이다. 지금은 비어 있으니 "아직 구현 전"이라는 가설을 뒷받침한다.
