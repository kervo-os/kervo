# 토큰 만료 버그, 어디부터 봐야 하나

## 결론부터

실제 코드(`h4-kit/repo`)에는 인증/토큰 관련 로직이 **전혀 존재하지 않는다.** `artifact-M9.md`의 "Known Decisions" 슬롯에 있는 "Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh" 항목은 저장소 코드로 뒷받침되지 않으며, 같은 문서 안에서도 다른 슬롯 항목이 실제 코드와 모순되는 정황이 있어 그대로 믿기는 위험하다. 그래서 "조사 시작점"은 코드의 특정 줄이 아니라, 먼저 "어떤 토큰을 말하는 것인지"와 "이 문서의 주장이 검증 가능한지"부터 확인하는 것이다.

## 확인한 사실 (코드 근거)

- `go.mod`: `module payments`, `go 1.23` 뿐, 의존성 0개. JWT 라이브러리(jwt-go, golang-jwt 등) 없음.
- 전체 저장소에서 `jwt|token|expir|auth`(대소문자 무시)로 grep한 결과, 매치는 `README.md`의 "card **tokenization**" 한 건뿐. 인증 토큰과 무관.
- `main.go`는 `func main() {}` + TODO 주석 한 줄, `handlers/charge.go`·`handlers/refund.go`·`store/db.go`·`queue/kafka.go` 전부 1~5줄짜리 스텁(로직 없이 주석/빈 함수)이다. 즉 이 checkout은 아직 스켈레톤 단계이고 라우팅·미들웨어 자체가 없다. `/auth/refresh` 같은 라우트도 존재하지 않는다.
- 커밋 로그도 이를 뒷받침: `f4ba12b initial service skeleton` → `bb2a9d7 add refund handler` → `54be496 stub sqlc migration` → `74d4646 kafka settlement + multi-region topology`. 인증 관련 커밋이 없다.

## artifact-M9.md 자체의 신뢰도 문제

"Known Decisions"는 문서 상단에 "slot sections carry trust-labeled observations"라고 명시되어 있어 확정 사실이 아니다. 실제로 같은 슬롯의 다른 두 항목이 코드와 정면으로 모순된다.

- "Settlement jobs go through the **legacy RabbitMQ** queue" → 그러나 `queue/kafka.go` 주석: "RabbitMQ was fully decommissioned in June 2026 — do not target it." Kafka 토픽 `settlement.v1` 사용 중. 심지어 같은 아티팩트의 "Repository Summary"(fact 섹션)는 정확히 "Kafka"라고 적혀 있어, **fact 섹션과 slot 섹션이 문서 내부에서 서로 충돌**한다.
- "Payments runs **single-region** in us-east-1; DR is cold-standby only" → 그러나 `deploy/regions.yaml`(2026-07 기준)은 us-east-1/eu-west-1/ap-northeast-2 3개 리전이 활성 상태이고, 최신 커밋명도 "kafka settlement **+ multi-region topology**"다.

두 항목이 명백히 stale인 것을 보면, 검증할 코드 근거가 전혀 없는 "JWT 15분 만료 / POST /auth/refresh" 항목도 stale이거나 이 저장소가 아닌 다른 서비스(예: 별도 API 게이트웨이/인증 서버)에 대한 설명일 가능성이 크다. 이 문서만으로는 확정할 수 없다.

## 조사 시작 순서 제안

1. **"토큰"의 정의부터 확정.** 버그 리포트/티켓이 말하는 게 (a) 카드 토큰화(vault 연동, README가 언급하는 "card tokenization") 인지, (b) 세션 인증용 JWT인지, (c) Kafka 컨슈머 오프셋이나 idempotency 토큰 같은 것인지 먼저 구분해야 한다. 이름이 겹쳐서 혼동되기 쉽다.
2. **인증 로직을 실제로 어느 저장소가 갖고 있는지 확인.** 이 `payments-api` checkout에는 auth 미들웨어/라우트가 없으므로, 버그가 이 서비스 자체 문제라면 해당 코드가 아직 구현 전이거나(스켈레톤 단계) 이 checkout에서 누락된 것이다. 게이트웨이/별도 auth 서비스 리포지토리가 따로 있는지부터 확인하는 게 우선이다.
3. **"Card PANs은 vault 서비스에만 저장" 결정도 이 repo 밖 얘기.** Known Decisions의 이 항목도 이 저장소 코드로는 검증 불가 — vault 관련 코드가 전혀 없다. 토큰화 관련 버그라면 vault 서비스 쪽 리포지토리를 봐야 할 가능성이 높다.
4. `handlers/charge.go`의 FIXME("idempotency keys not enforced")를 "token" 문제로 잘못 지칭하고 있는 건 아닌지 확인. 명칭 혼용 가능성 체크.
5. `store/db.go`가 스텁(`TODO: migrate to sqlc`)이라 세션/토큰 저장 테이블 존재 여부 자체가 불명. DB 스키마가 실제로 어디 정의돼 있는지(이 repo엔 마이그레이션 파일 없음) 확인 필요.
6. **아티팩트 주장의 출처 추적.** "Auth is JWT..." 문구가 어디서 왔는지(팀 위키, 다른 서비스의 CLAUDE.md, 과거 PR 설명 등) 원출처를 찾아 검증한 뒤에만 신뢰 여부를 판단할 것. `kervo compile`로 자동 생성된 문서이므로 사람이 검증하지 않은 추론일 수 있다.

## 확실하지 않은 것 (명시)

- "토큰 만료 버그"가 정확히 어떤 토큰(인증 JWT / 카드 토큰화 / 메시지 토큰)을 가리키는지 — 티켓 원문 없이는 확정 불가.
- JWT 15분 만료 및 `POST /auth/refresh` 결정이 이 `payments-api` 저장소에 실제로 적용되는지 — 코드 근거 전무, 문서 내부 모순 정황으로 볼 때 신뢰도 낮음.
- 이 checkout이 전체 서비스 코드인지, 스켈레톤/발췌본이라 실제 배포 코드는 별도에 있는지 — 저장소 규모(파일 9개, 대부분 1~5줄 스텁)로 보아 후자일 가능성이 있으나 단정할 근거는 없음.
