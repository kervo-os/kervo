## 0. 이 문서의 성격과 신뢰도 한계 (먼저 읽어주세요)

- 이 초안은 `artifact-M9.md`(컨텍스트 아티팩트)와 `repo/` 실제 코드를 근거로 작성했습니다.
- 저장소는 아직 스켈레톤 단계입니다(`main.go`, `handlers/*.go`, `store/db.go`가 대부분 주석 한 줄짜리 스텁). 로드밸런서, DNS/GSLB, DB 복제, 모니터링/온콜 설정 등 실제 인프라 구성 파일이 저장소 안에 없습니다. 따라서 이 문서의 상당 부분은 "확인 필요(TBD)"로 남겨두었고, 임의로 수치나 메커니즘을 지어내지 않았습니다.
- **아티팩트와 코드가 서로 다른 말을 하는 항목이 있습니다.** 이 경우 더 최근 시점·더 구체적인 코드 근거를 우선했고, 아래 1장에서 명시적으로 정정합니다.

## 1. 아티팩트 vs 코드 불일치 정정 (중요, 먼저 확인)

`artifact-M9.md`의 "Known Decisions" 슬롯에는 다음 두 항목이 있습니다.

- "Payments runs single-region in us-east-1; DR is cold-standby only."
- "Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there."

그런데 실제 코드는 이와 반대입니다.

- `deploy/regions.yaml` (주석: "active deployment topology (July 2026)"): `us-east-1`, `eu-west-1`, `ap-northeast-2` 3개 리전이 명시되어 있습니다. → 단일 리전이 아니라 **멀티 리전**입니다.
- `queue/kafka.go`: "Settlement jobs are produced to the Kafka topic `settlement.v1`. RabbitMQ was fully decommissioned in June 2026 — do not target it." → **RabbitMQ는 2026년 6월에 완전히 폐기**되었고 현재는 **Kafka**(`settlement.v1` 토픽)를 사용합니다.
- `README.md`도 "Settlement jobs flow through Kafka (queue/); deployed multi-region (deploy/regions.yaml)"라고 명시해 위 코드 근거와 일치합니다.
- 최근 변경 이력(`74d4646`, 2026-07-04, "kafka settlement + multi-region topology")도 이 전환이 최근에 일어났음을 보여줍니다.

**결론**: 아티팩트의 해당 두 "Known Decisions"는 stale(구버전) 정보이며, 실제로는 폐기된 RabbitMQ를 "새 컨슈머가 바인딩해야 할 대상"으로 잘못 안내하고 있어 그대로 따르면 위험합니다. 이번 계획은 코드 근거(멀티 리전 3곳 + Kafka)를 기준으로 작성했습니다. 아티팩트는 재생성(`kervo compile`) 또는 수동 정정이 필요해 보입니다(이 문서 범위 밖이라 별도 요청 필요).

아티팩트의 나머지 Known Decisions(PAN은 vault 서비스에만 저장 / JWT 15분 만료 인증 / 환불은 provider webhook으로 비동기 처리)는 저장소 코드로 직접 확인도, 반박도 되지 않습니다(관련 구현 파일이 없거나 스텁뿐). 반박 근거가 없으므로 잠정적으로 유효하다고 가정했지만, "코드로 검증됨"이 아니라 "문서 주장, 미검증" 상태로 표시해 둡니다.

| 항목 | 확인된 내용 | 근거 |

**확인 불가(코드에 없음, 인프라 쪽 확인 필요)**:
- 3개 리전이 active-active인지 active-passive(콜드/웜 스탠바이)인지
- 리전 간 트래픽 라우팅/failover 메커니즘(DNS 가중치, GSLB, Anycast, LB 헬스체크 등)
- Kafka가 리전별 독립 클러스터인지, 리전 간 복제(MirrorMaker2 등)가 있는지, 있다면 오프셋/순서 보장 방식
- DB가 리전별 분리인지 글로벌 단일 프라이머리인지, 복제 지연 및 일관성 모델
- vault 서비스 자체의 리전 이중화 여부
- 모니터링/알림/온콜 도구와 에스컬레이션 정책

이 항목들은 이번 초안에서 임의로 가정하지 않았습니다. 실제 대응 절차(3장)의 상당 부분은 이 정보가 확정되어야 구체화됩니다.

## 3. 장애 시나리오별 초기 대응 방향

### 3.1 단일 리전 완전 장애 (예: eu-west-1 다운)
- 해당 리전으로의 신규 트래픽 라우팅 차단 필요 — **단, 실제 라우팅 계층/방식이 미확인이므로 구체적 명령/콘솔 절차는 인프라팀 확인 후 러너북에 채워야 함.**
- Kafka: 장애 리전에 붙어있던 프로�듀서/컨슈머 처리 확인. 리전 간 복제 여부가 불명확하므로, 장애 리전에서 처리 중이던 정산 이벤트의 유실 가능성을 배제하지 말 것.
- 다른 두 리전이 트래픽을 흡수할 수 있는 용량(capacity)을 갖는지 사전 확인 필요(현재 리소스/오토스케일링 설정 근거 없음).

### 3.2 Kafka(`settlement.v1`) 장애/지연
- 정산 배치가 지연되거나 유실될 위험. `main.go`의 settlement cron이 아직 연결되지 않은 상태(TODO)라, 실제 운영 환경에서 크론이 이미 구현되어 있는지부터 확인 필요.
- 재처리 시 idempotency key가 강제되지 않는 상태(`handlers/charge.go` FIXME)이므로, 큐 재처리/재시도 과정에서 **중복 정산·중복 청구 위험**이 큼. 장애 대응 중 재시도를 걸기 전에 반드시 수동 중복 검증 절차를 넣을 것을 권고.
- 아티팩트의 Known Risks에도 "정산 크론에 재시도 로직이 없는 것으로 보임"이라는 지적이 있음 — 이 역시 재확인 필요.

### 3.3 DB/스토어 장애
- `store/db.go`가 스텁 상태라 실제 DB 엔진, 복제 구성, 백업/복구(RPO에 직결) 정보가 없음. 이 시나리오는 세부 절차를 채우기 전에 반드시 실제 DB 인프라 문서를 확보해야 함.

### 3.4 Vault 서비스(카드 PAN 조회) 리전 장애
- vault가 별도 서비스로 분리되어 있다는 것만 확인되고(코드 근거는 없음, 아티팩트 주장), 이중화 여부는 불명. Vault 접근 불가 시 결제/환불 흐름이 어떻게 저하되는지 별도 확인 필요. PAN을 API 레이어에 절대 캐시하지 않는다는 팀 규칙(`CLAUDE.md`: "Never log PANs")은 장애 대응 중 임시 로깅/디버깅을 할 때도 반드시 지켜야 함.

## 4. 장애 대응에 영향을 주는 기존 코드 리스크 (해결 전 상태로 대응해야 함)

- **idempotency 미강제** (`handlers/charge.go` FIXME) — 장애/재시도 상황에서 가장 위험한 항목. 리전 페일오버 중 클라이언트 재시도나 컨슈머 재처리가 겹치면 중복 청구로 직결됨.
- **정산 크론 미연결** (`main.go` TODO) — 정산이 실제로 자동 수행되는지조차 이 저장소 기준으로는 불확실.
- **정산 재시도 로직 부재 추정** (아티팩트 Known Risks) — 장애 후 재처리 전략을 별도로 설계해야 함.
- **DB sqlc 마이그레이션 진행 중** (`store/db.go` TODO) — 데이터 계층이 과도기 상태.

권고: 위 4개 항목 중 최소한 idempotency 강제는 지역 장애 대응 계획을 "운영 가능"으로 승인하기 전에 반드시 해결하거나, 해결 전까지는 장애 시 자동 재시도를 금지하고 수동 검토를 의무화하는 임시 가드레일을 둘 것.

## 5. 대응 절차 초안 (일반형, 다수 단계는 세부 확정 필요)

1. **감지**: 모니터링/알림 출처 미확인 — 확인 후 채움.
2. **트리아지/에스컬레이션**: 온콜 정책 미확인 — 확인 후 채움. 영향 리전, 영향 컴포넌트(정산/결제/환불) 최초 분류.
3. **트래픽 격리**: 장애 리전을 라우팅에서 제외. 구체 메커니즘 미확인(3.1 참고).
4. **정산 파이프라인 안전 정지/격리**: Kafka 컨슈머를 무조건 재가동하기 전에, idempotency 미강제 상태를 감안해 중복 처리 여부를 수동 확인.
5. **데이터 정합성 점검**: 장애 리전에서 처리 중이던 트랜잭션의 유실/중복 여부 확인. PAN 미노출(vault 경계 유지) 재확인.
6. **복구/페일백**: 리전 복구 후 트래픽 점진 복귀, 정산 큐 재동기화 확인.
7. **사후보고(post-mortem)**: 근본 원인, 재발 방지, 이번 문서의 TBD 항목 업데이트.
