# PAN 로깅 감사 체크리스트 — payments-api

## 0. 전제 및 스코프

- 근거 문서: `artifact-M9.md` (kervo 컨텍스트 아티팩트), 근거 코드: `h4-kit/repo` (payments-api, Go, 커밋 4개 규모의 초기 스켈레톤).
- **중요한 전제 조건**: 현재 저장소는 `main.go`, `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `queue/kafka.go` 모두 한 줄짜리 패키지 선언 + 주석(TODO/FIXME) 수준의 **스텁 코드**이며, 실제 로깅 호출부(`log.*`, `fmt.Printf`, 구조화 로깅 라이브러리 등)가 **한 줄도 존재하지 않는다.** `go.mod`에도 로깅 라이브러리 의존성이 없다.
- 따라서 이 체크리스트는 "현재 코드에서 PAN 로깅 위반을 발견했다"는 감사가 아니라, **① 문서(artifact)상 주장과 코드 사실의 불일치를 먼저 정리**하고 **② 앞으로 charge/refund/settlement 로직이 채워질 때 PAN이 로그에 노출되지 않도록 지켜야 할 체크포인트**를 정의하는 것에 목적이 있다. 이 점을 감사 보고서 상단에 명시할 것을 권고한다.
- 확인된 유일한 명문 규칙: `CLAUDE.md` — "Never log PANs." (강제 메커니즘은 코드/CI 어디에도 없음 — 아래 6번 항목 참조)

## 1. 문서-코드 불일치 (감사 스코프에 영향을 주는 전제 오류)

아티팩트의 "Known Decisions" 슬롯 중 아래 두 항목은 실제 코드와 상충하므로, PAN 로깅 감사 스코프를 잡을 때 아티팩트를 그대로 신뢰해서는 안 된다.

- [ ] **큐 시스템 불일치**: 아티팩트는 "Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there"라고 명시하지만, 실제 `queue/kafka.go`의 코드 주석은 "RabbitMQ was fully decommissioned in June 2026 — do not target it"이며 실제 사용 큐는 Kafka 토픽 `settlement.v1`이다. → **감사 시 RabbitMQ가 아니라 Kafka 파이프라인(프로듀서/컨슈머/DLQ/브로커 로그)을 대상으로 잡아야 한다.** 아티팩트의 해당 결정 항목은 stale로 판단됨.
- [ ] **리전 토폴로지 불일치**: 아티팩트는 "Payments runs single-region in us-east-1; DR is cold-standby only"라고 명시하지만, `deploy/regions.yaml`(2026년 7월 기준 "active deployment topology")은 `us-east-1`, `eu-west-1`, `ap-northeast-2` 3개 리전을 나열하고 있어 **활성 멀티리전 운영**을 시사한다. → PAN 로깅 감사는 단일 리전이 아니라 **3개 리전 각각의 로그 수집/집계 파이프라인(중앙 로그 저장소, 리전 간 복제, 데이터 레지던시)을 모두 포함**해야 한다.
- [ ] 위 두 불일치가 존재하는 상태에서, 같은 슬롯에 있는 "Card PANs are stored only in the vault service; the API layer never persists them"라는 결정 항목도 **코드로 검증되지 않은 주장(unverified assertion)**으로 취급해야 한다. 저장소 어디에도 vault 연동 코드, 토큰화 로직, PAN 필드 정의가 없다 (전체 저장소에 대해 `pan|card|log|mask|redact|token|vault` 키워드로 검색한 결과 README 문구와 CLAUDE.md 규칙 문구 외 실제 코드 매치 없음).

## 2. 코드 레벨 로깅 지점 체크리스트 (구현 시점 기준으로 적용)

### A. 결제/환불 요청 처리 (`handlers/charge.go`, `handlers/refund.go`)
- [ ] 요청 바디 전체를 로깅하는 미들웨어(access log, request/response dump)가 PAN, CVV, 만료일 등 카드 데이터 필드를 그대로 포함하지 않는지 확인.
- [ ] 에러/예외 발생 시 스택트레이스나 에러 메시지에 요청 페이로드(raw request struct)를 그대로 덤프하지 않는지 확인 — Go에서 `%+v`, `%v`로 struct를 그대로 로그에 찍는 패턴이 특히 위험.
- [ ] `handlers/charge.go`의 FIXME "idempotency keys not enforced" 항목이 구현될 때, 멱등키 생성 로직이 카드 데이터(PAN 전체/부분)를 해시 원문으로 사용하면서 원문을 디버그 로그에 남기지 않는지 확인. 멱등키 충돌 로그에 원본 요청을 첨부하지 않는지 확인.
- [ ] 결제 실패/거절(decline) 응답을 처리·로깅할 때 PSP(결제대행사) 응답 원문 그대로를 로그에 남기지 않는지 확인 (PSP 응답에 마스킹된 PAN이 포함되는 경우가 흔함 — 마스킹 형식이 유지되는지, 재마스킹/재구성 위험은 없는지 확인).
- [ ] `refund.go`는 "refunds are processed asynchronously via provider webhooks"(아티팩트 주장)이라는데, 웹훅 수신 핸들러가 구현되면 웹훅 payload 전체를 audit log/디버그 로그에 그대로 남기지 않는지 확인 (webhook 재시도 실패 시 payload를 로그/DLQ에 평문 저장하는 패턴 주의).

### B. 메시징/큐 (`queue/kafka.go`, 토픽 `settlement.v1`)
- [ ] Kafka 프로듀서가 정산 메시지에 PAN 또는 카드 원본 데이터를 직렬화해서 넣지 않는지 확인 (정산에는 토큰/거래ID만 필요해야 함).
- [ ] 컨슈머 측 처리 실패 시 메시지 내용을 그대로 에러 로그/DLQ에 적재하는지 확인 — DLQ도 "로그"와 동일한 감사 대상으로 취급.
- [ ] Kafka 브로커/클라이언트 라이브러리 자체의 디버그 로그 레벨(예: consumer group rebalance 시 메시지 샘플 출력 등)이 프로덕션에서 비활성화되어 있는지 확인.
- [ ] RabbitMQ 데코미션(2026년 6월) 이후에도 과거 RabbitMQ 관련 로그/브로커 디스크에 PAN이 남아있지 않은지 별도 확인 (레거시 인프라 잔존 데이터 폐기 여부) — 아티팩트가 여전히 RabbitMQ를 최신 결정으로 기록하고 있었다는 점에서, 데코미션 이후 정리(로그 삭제/보존기간 만료) 절차가 실제로 수행됐는지 재확인 필요.

### C. 데이터 저장 계층 (`store/db.go`)
- [ ] `store/db.go`의 TODO "migrate to sqlc"가 진행될 때 생성되는 쿼리 로깅(ORM/드라이버의 SQL 쿼리 로그, 예: `database/sql` 드라이버의 쿼리 트레이싱)이 바인딩 파라미터 값(PAN 등)을 평문으로 출력하지 않는지 확인. sqlc/드라이버의 쿼리 로그 레벨이 프로덕션에서 파라미터 값을 마스킹하는지 점검.
- [ ] DB 스키마 설계 시 PAN 컬럼 자체가 API 계층 DB에 존재하지 않는지 확인 (아티팩트 주장대로 vault에만 저장된다면, payments DB에는 토큰/참조ID만 있어야 함 — 스키마가 확정되면 이 항목을 재검증).
- [ ] 슬로우 쿼리 로그(slow query log)에 쿼리 파라미터가 포함되도록 설정되어 있지 않은지 확인.

### D. 애플리케이션 부트스트랩/설정 (`main.go`)
- [ ] `main.go`의 TODO "wire settlement cron"이 구현될 때, cron 실행 로그가 배치 대상 거래의 카드 데이터를 요약/샘플로 남기지 않는지 확인.
- [ ] 아티팩트가 지적한 리스크("Settlement cron appears to lack retry logic")가 해소되면서 retry/backoff 로직이 추가될 때, 재시도 실패 로그에 원본 정산 payload(PAN 포함 가능성)를 그대로 남기지 않는지 확인 — 재시도 로직과 로깅은 함께 리뷰 필요.

## 3. 인프라/플랫폼 레벨 체크리스트

- [ ] (멀티리전 확인됨 — 위 1번 참조) `us-east-1`, `eu-west-1`, `ap-northeast-2` 각 리전의 로그 수집기(예: 사이드카, 중앙 로그 집계 시스템)가 동일한 PAN 마스킹/필터링 정책을 적용하는지 확인 — 리전별로 로깅 파이프라인 구성이 다르면 특정 리전에서만 마스킹이 누락될 수 있음.
- [ ] 리전 간 로그 복제/중앙화 시 전송 경로(TLS 여부)와 저장소(장기보관 여부, 보존기간)를 확인 — PAN이 실수로 로그에 유입된 경우 확산 범위가 3개 리전으로 커짐.
- [ ] 로드밸런서/API 게이트웨이/WAF의 액세스 로그가 요청 바디 전체를 캡처하도록 설정되어 있지 않은지 확인 (인프라 레벨 로깅은 애플리케이션 코드 감사만으로 놓치기 쉬움).
- [ ] APM/트레이싱 도구(예: 분산 트레이싱의 span 속성)에 HTTP 요청/응답 바디가 자동 캡처되도록 설정되어 있지 않은지 확인.
- [ ] 로그 저장소(예: 중앙 로그 저장 시스템) 자체의 접근 통제 및 PAN 스캐닝(사후 탐지) 도구가 존재하는지 확인 — 사전 마스킹 실패에 대한 2차 방어선.

## 4. 서드파티/외부 연동 체크리스트

- [ ] PSP(결제대행사) 연동 SDK/클라이언트 라이브러리의 디버그 모드가 프로덕션에서 꺼져 있는지 확인 (많은 PSP SDK가 디버그 모드에서 요청/응답 전체를 stdout에 출력).
- [ ] 환불 웹훅(refund provider webhook) 수신부의 로깅 — 웹훅 서명 검증 실패, 파싱 실패 시 원본 바디를 에러 로그에 남기는 관행이 흔하므로 별도 확인.
- [ ] vault 서비스와의 통신(아티팩트 주장상 PAN이 저장되는 유일한 곳) — vault 클라이언트 호출 시 요청/응답을 로깅하는 코드가 있다면 PAN이 vault 왕복 과정에서 로그에 노출될 수 있음. vault 연동 코드가 아직 없으므로, **구현 시점에 최우선으로 리뷰 대상에 포함**할 것을 권고.

## 5. 거버넌스/프로세스 체크리스트 (정책 강제 메커니즘)

- [ ] `CLAUDE.md`의 "Never log PANs" 규칙이 **자동으로 강제되는 수단이 현재 전혀 없음**을 확인함 (lint rule, pre-commit hook, CI 정적분석, 로그 필드 스캐닝 도구 등 모두 부재). 이 규칙은 현재 사람이 리뷰 시점에 기억해야만 지켜지는 상태 — 감사 결과에 "정책은 문서화되어 있으나 자동 강제 수단 없음"을 리스크로 명시할 것.
- [ ] PR/코드 리뷰 체크리스트에 "새로 추가된 로그 문장에 PAN/카드 데이터가 포함되는가"를 명시적 항목으로 넣을 것을 권고 (현재 CLAUDE.md에 언급된 팀 컨벤션은 "테이블 드리븐 테스트" 뿐, 로깅 리뷰 관련 컨벤션 없음).
- [ ] 정적분석 도구(예: 정규식 기반 PAN 패턴 탐지 — Luhn 알고리즘 매칭 등)를 CI에 도입해 로그 출력 코드 경로를 스캔하는 방안 검토.
- [ ] PCI-DSS Requirement 3.4(저장 시 PAN 비식별화) / 10.x(로깅 및 모니터링) 매핑이 필요한 경우, 이는 조직의 실제 PCI-DSS 준수 범위 판단(QSA 확인)이 필요한 사항으로, 이 저장소 분석만으로는 확정할 수 없음 — **불확실 항목으로 별도 표시.**

## 6. 검증 불가 / 불확실 항목 (명시적 caveat)

아래는 코드로 확인되지 않았거나 정보가 부족해 "확실하다"고 말할 수 없는 항목이다.

- **불확실**: "PAN은 vault 서비스에만 저장되고 API 계층은 절대 영속화하지 않는다" — 이는 아티팩트의 주장(Known Decision)일 뿐, 코드/스키마 어디에도 이를 구현하거나 강제하는 근거가 없다. 스텁 상태이므로 향후 구현이 이 주장과 다르게 진행될 가능성을 배제할 수 없음.
- **불확실**: "Auth는 JWT 15분 만료" — 인증 관련 코드가 저장소에 존재하지 않아 확인 불가. PAN 로깅과 직접 관련은 낮지만, 인증 실패 로그에 토큰/자격증명이 남는지도 향후 확인 필요.
- **불확실**: 아티팩트의 리전 토폴로지("single-region us-east-1")와 실제 `regions.yaml`(3개 리전) 중 어느 쪽이 "현재 운영 사실"에 더 가까운지는 이 저장소만으로 최종 판단 불가 — `regions.yaml`이 배포 목표(desired state) 파일일 수도, 아티팩트가 stale일 수도 있다. 실제 운영 인프라(클라우드 콘솔/IaC 상태) 확인이 필요.
- **불확실**: PCI-DSS 등 규제 준수 범위 — 조직 차원의 컴플라이언스 문서가 이 코드 저장소 범위 밖에 있을 가능성이 높아, 이 감사만으로 규제 준수 여부를 판단할 수 없음.
- **확인됨(참고)**: 저장소 자체에는 실제 로깅 호출이 전혀 없으므로, "현재 시점에 PAN이 로그에 남고 있다"는 위반 사실은 **발견되지 않았다** (구현이 없으니 위반도 없음). 이 체크리스트는 사전(preventive) 감사 성격임을 재강조.

## 7. 권고 우선순위 (요약)

1. (High) 아티팩트의 큐/리전 관련 Known Decisions를 실제 코드 기준으로 정정 — 잘못된 전제로 후속 개발/감사가 진행되는 것을 방지.
2. (High) charge/refund/settlement 로직 구현 착수 전에 "로깅 표준"(구조화 로깅 라이브러리 선정 + PAN 필드 마스킹 규칙)을 먼저 정의 — 현재 go.mod에 로깅 라이브러리조차 없어 백지 상태.
3. (Medium) CI/PR 단계에 PAN 패턴 자동 탐지 도입 검토.
4. (Medium) 멀티리전(3개 리전) 로그 파이프라인의 마스킹 정책 동일성 확인.
5. (Low) vault 연동 및 PSP/웹훅 연동 코드 작성 시 위 4번 섹션 체크리스트를 리뷰 게이트에 포함.
