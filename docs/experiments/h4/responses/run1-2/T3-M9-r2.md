- **저장소(repo) 안에 "토큰"과 관련된 코드가 전혀 없다.** `jwt|auth|token|expir|refresh` 전체 grep 결과 실제 코드에서 매치되는 곳은 0건이고, README의 "card tokenization"이라는 단어 한 줄이 유일한 히트다.

## 2. 아티팩트("Known Decisions")를 그대로 믿으면 안 되는 이유

`artifact-M9.md`의 Known Decisions 슬롯:

```
- Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there.
- Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh.
```

첫 번째 문장(RabbitMQ)은 실제 코드와 **정면으로 충돌**한다 — `queue/kafka.go`는 "RabbitMQ was fully decommissioned in June 2026 — do not target it"라고 명시하고, 대신 Kafka 토픽 `settlement.v1`을 쓴다. 게다가 가장 최근 커밋(`74d4646`, 2026-07-04)이 정확히 "kafka settlement + multi-region topology"다. 즉 이 Known Decisions 슬롯은 **이 최근 커밋 이전 시점의 낡은(stale) 관찰**이고, 아직 재생성(`kervo compile`)되지 않아 코드 변경을 반영하지 못한 상태다. 아티팩트 헤더에도 "slot sections carry trust-labeled observations" (사실 확정이 아니라 신뢰도 라벨이 붙은 관찰일 뿐)이라고 명시돼 있다.

같은 슬롯, 같은 위험도로 적힌 두 번째 문장("Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh")도 **똑같이 낡았거나, 이 리포와 무관한 다른 서비스에 대한 관찰일 가능성**을 배제할 수 없다. 실제로 이 문장을 뒷받침하는 코드가 리포 안에 전혀 없으므로(위 grep 결과), 이걸 "이 리포 어딘가에 15분 만료 JWT 로직이 있다"는 사실로 취급해서 코드를 뒤지면 시간 낭비다.

## 3. "토큰"이 뭘 가리키는지부터 확인 필요 (전제가 불확실함)

이 프로젝트에서 "토큰"은 최소 두 가지 의미로 쓰일 수 있고, 버그 리포트가 어느 쪽인지 이 문서만으로는 확정할 수 없다:

1. **인증(Auth) 토큰(JWT, 세션)** — 아티팩트가 언급하는 "15분 만료 + /auth/refresh".
2. **카드 tokenization** — README 1줄: "Internal payments service: card tokenization and settlement batching." PCI 카드 토큰(카드 번호를 대체하는 vault 토큰)의 만료/재발급 문제일 수도 있음. 아티팩트 Known Decisions에도 "Card PANs are stored only in the vault service; the API layer never persists them"이라는 문장이 있어, 카드 토큰 발급/만료 로직도 **vault 서비스라는 별도 서비스**에 있을 가능성이 크다.

둘 다 이 저장소(`payments`) 안에는 구현 코드가 없다 — 인증도, vault 연동도 이 리포에는 없다.

## 4. 결론 및 다음 액션

**확실한 것:**
- 이 리포(payments-api) 안에는 토큰(인증이든 카드든) 관련 코드가 0건이다. `main.go`는 빈 함수, 핸들러 2개는 스텁, 의존성도 0개.
- 코드로 검증 가능한 유일한 "토큰" 관련 흔적은 README의 "card tokenization" 단어 하나뿐.

**확실하지 않은 것 (전제로 깔면 안 되는 것):**
- 아티팩트의 "JWT 15분 만료 / POST /auth/refresh" 결정이 지금도 유효한지 — 옆 항목(RabbitMQ)이 이미 틀렸다고 코드로 확인됐으므로 신뢰도가 낮다.
- 버그 리포트가 말하는 "토큰"이 인증 토큰인지 카드 tokenization 토큰인지.
- 인증/vault 로직이 별도 리포(별도 서비스)에 있는지 여부 — 이 워크스페이스에서는 확인 불가(해당 리포가 안 보임).

**시작 지점 제안, 우선순위 순:**
1. 버그 리포트/티켓 원문을 다시 확인해서 "토큰"이 인증 토큰인지 카드 토큰인지 먼저 확정한다 (이 리포만 봐서는 판단 불가).
2. 인증 토큰 문제라면, **이 리포에는 코드가 없으므로** 실제 JWT 발급·검증을 담당하는 별도의 auth/identity 서비스 리포를 찾아야 한다. `payments-api`는 그 토큰을 소비(consume)만 하는 쪽일 가능성이 있는데, 그마저도 미들웨어 코드가 아직 없어 검증 불가.
3. 카드 tokenization 문제라면, vault 서비스(별도 리포로 추정) 쪽을 봐야 한다. 이 리포의 결정 사항("API 계층은 PAN을 절대 저장하지 않음")을 보면 tokenization 로직 자체가 vault 서비스 소유다.
4. 만약 정말로 이 리포 안에서 뭔가를 봐야 한다면, 유일하게 연관 지을 만한 실마리는 `deploy/regions.yaml`의 최근 멀티리전 확장(us-east-1 → +eu-west-1, ap-northeast-2, 커밋 `74d4646`)이다 — 리전 간 시계 스큐(clock skew)나 세션/토큰 검증이 리전별로 어긋나는 문제일 가능성이 이론적으로는 있지만, 이는 순수 추측이며 뒷받침하는 코드나 로그가 전혀 없다.
5. 팀에 "Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh" 결정 문장의 출처와 최신성을 직접 확인 요청 — 이 리포의 소스가 아니라 사람 기억/구두 결정에서 왔을 가능성이 있음.

요약하면: **코드 기반으로는 "여기서부터 보라"고 가리킬 파일이 없다.** 이 리포는 아직 인증/토큰화 계층이 구현되지 않은 스켈레톤이고, 아티팩트가 제시하는 JWT 관련 정보는 검증되지 않았을 뿐 아니라 같은 섹션의 다른 정보가 코드와 이미 모순된 전례가 있어 신뢰하기 어렵다. 먼저 "토큰"의 정확한 의미와 실제 구현 위치(다른 서비스/리포일 가능성)를 확인하는 것이 코드를 뒤지는 것보다 우선이다.
