# PAN 로깅 감사 체크리스트 — payments-api

## 0. 조사 방법 및 중요 전제

이 체크리스트는 다음을 근거로 작성했다.

1. `artifact-K7.md` (kervo가 생성한 컨텍스트 아티팩트, fact 섹션 + slot 섹션)
2. 저장소 실제 코드 전체 (`main.go`, `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `queue/kafka.go`, `deploy/regions.yaml`, `README.md`, `CLAUDE.md`, `go.mod`, git log)
3. PCI DSS 등 결제 카드 산업 일반 상식 (저장소 안에는 근거가 없는, 외부 지식)

**먼저 밝혀야 할 전제 두 가지:**

- **이 저장소는 실질적으로 스켈레톤 상태다.** `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `main.go`는 전부 주석 한 줄(TODO/FIXME)뿐이고 실제 로깅 호출, HTTP 핸들러 본문, DB 쿼리, 카드 데이터 처리 로직이 전혀 없다. 즉 "지금 코드에 PAN 로깅 버그가 있는지"는 검증할 대상 자체가 존재하지 않는다. 이 체크리스트는 **향후 구현될 코드를 게이트하기 위한 사전 감사 체크리스트**로 이해해야 한다.
- 아티팩트의 "Known Decisions" 슬롯에 있는 "Card PANs are stored only in the vault service; the API layer never persists them"는 **미검증 관찰(slot)**이다. 아티팩트 자체가 "slot sections carry trust-labeled observations"라고 명시하며, 이 저장소에는 vault 연동 코드가 전혀 없어 실제로 확인할 수 없다. 반대로 같은 아티팩트의 "Deprecated / Stale Notes" 슬롯은 "정산은 레거시 RabbitMQ를 사용한다"고 적혀 있는데, 실제 `queue/kafka.go` 코드는 "RabbitMQ was fully decommissioned in June 2026 — do not target it"라고 명시하여 **정면으로 상충**한다. 이는 슬롯 섹션을 그대로 신뢰해서는 안 된다는 실증 사례이므로, 아래 체크리스트에서도 슬롯 근거 항목은 "미확인"으로 별도 표기했다.
