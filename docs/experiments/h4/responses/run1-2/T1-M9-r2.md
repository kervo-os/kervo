# payments-api 정산 잡 큐잉 개선 — 코드 스케치

## 0. 이 문서의 성격

- `artifact-M9.md`와 `repo/` 실제 코드를 근거로 작성했습니다. 저장소는 스켈레톤 단계이고(`main.go`, `handlers/*.go`, `store/db.go`가 대부분 주석 한 줄짜리 스텁, `queue/kafka.go`는 함수 시그니처만 있고 본문이 비어 있음), Kafka 클라이언트 라이브러리도 `go.mod`에 아직 없습니다(`module payments` / `go 1.23`뿐, 의존성 0개). 따라서 아래는 실제로 컴파일/실행 가능한 완성 코드가 아니라 **설계 방향을 보여주는 스케치**이며, 확정되지 않은 부분은 각 절에서 명시적으로 "가정/불확실"로 표시했습니다.

## 1. 아티팩트 vs 코드 불일치 정정 (먼저 확인 필요)

`artifact-M9.md`의 **Known Decisions** 슬롯에 다음 두 항목이 있습니다.

- "Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there."
- "Payments runs single-region in us-east-1; DR is cold-standby only."

그런데 실제 코드는 반대를 말합니다.

- `queue/kafka.go`: "Settlement jobs are produced to the Kafka topic `settlement.v1`. RabbitMQ was fully decommissioned in June 2026 — do not target it."
- `README.md`: "Settlement jobs flow through Kafka (queue/); deployed multi-region (deploy/regions.yaml)."
- `deploy/regions.yaml` (주석 "active deployment topology (July 2026)"): `us-east-1`, `eu-west-1`, `ap-northeast-2` 3개 리전.
- 최근 커밋 `74d4646` (2026-07-04, "kafka settlement + multi-region topology")도 이 전환을 뒷받침합니다.

**결론**: 아티팩트의 위 두 Known Decisions는 stale 정보로 판정하고 무시했습니다. 이번 스케치는 **Kafka + 멀티 리전(3개)** 전제로 작성했습니다. RabbitMQ를 타깃으로 하는 코드는 만들지 않았습니다. (참고로 Doc Summaries 슬롯의 "table-driven tests 컨벤션"도 실제 `CLAUDE.md`(내용: "Never log PANs" 한 줄뿐)에는 없는 내용이라 슬롯 섹션 전반을 신뢰하지 않고 코드/README 1차 근거만 채택했습니다.)

아티팩트 **Known Risks**의 "정산 크론에 재시도 로직이 없어 보임"은 `main.go`(`// TODO: wire settlement cron`이 전부)로 실제 확인되므로 이 부분은 그대로 채택해 설계에 반영했습니다.

## 3. 설계 방향

1. **애플리케이션 레벨 idempotency를 1순위로 둔다.** Kafka 브로커의 "idempotent producer" 기능은 클라이언트 라이브러리마다 지원 범위가 다르고(아래 5절 참고), 크론 재실행·DLQ 재처리·수동 replay 같은 상위 레벨 중복까지는 못 막습니다. 그래서 정산 배치 단위(merchant × 시간창)마다 결정적 idempotency key를 만들고, DB에 "claim" 행을 선점(insert-if-absent)한 뒤에만 발행하도록 합니다.
2. **재시도 + DLQ.** 아티팩트/코드 모두가 지적하는 "재시도 로직 없음" 문제를 producer 쪽에 지수 백오프 재시도 + 실패 시 `settlement.v1.dlq` 토픽으로 라우팅하는 방식으로 메꿉니다. 유실 대신 사람이 볼 수 있는 곳에 쌓이게 합니다.
3. **정렬/파티셔닝.** 메시지 키를 idempotency key(사실상 merchant 단위)로 잡아 같은 merchant의 정산 이벤트 순서를 파티션 내에서 보존합니다.
4. **PAN 미노출.** `CLAUDE.md` 팀 규칙("Never log PANs")에 따라 정산 잡 payload와 로그에는 PAN/카드 원본 데이터를 절대 넣지 않고, merchant ID·금액·기간만 다룹니다.
5. **charge.go의 idempotency FIXME는 여기서 대신 해결되지 않는다는 점을 명시.** 이 스케치의 dedupe는 "같은 정산 배치를 두 번 발행하지 않는다"만 보장하며, 애초에 중복 charge 레코드가 store에 쌓이는 문제(charge.go FIXME)는 별도로 고쳐야 합니다. 두 문제를 섞으면 "정산은 안 겹치는데 원장 자체가 중복"인 상태가 될 수 있습니다.

## 4. 코드 스케치

### 4.1 `queue/job.go` (신규)

```go
package queue

import (
	"crypto/sha256"
	"encoding/hex"
	"fmt"
	"time"
)

// SettlementJob is one settlement batch unit published to Kafka topic "settlement.v1".
//
// IMPORTANT: never add PAN or any raw card data here (CLAUDE.md: "Never log PANs").
// Only merchant/settlement identifiers and amounts belong in this struct.
type SettlementJob struct {
	IdempotencyKey string    `json:"idempotency_key"`
	MerchantID     string    `json:"merchant_id"`
	Region         string    `json:"region"` // us-east-1 | eu-west-1 | ap-northeast-2
	WindowStart    time.Time `json:"window_start"`
	WindowEnd      time.Time `json:"window_end"`
	AmountMinor    int64     `json:"amount_minor"` // minor unit (cents) — avoid floats for money
	Currency       string    `json:"currency"`
	Attempt        int       `json:"attempt"`
	CreatedAt      time.Time `json:"created_at"`
}

// NewIdempotencyKey is deterministic for (merchant, window), so a cron re-run,
// a DLQ replay, or a manual retrigger of the same batch always yields the same
// key and can be deduped against store state before publishing.
func NewIdempotencyKey(merchantID string, windowStart, windowEnd time.Time) string {
	h := sha256.Sum256([]byte(fmt.Sprintf("%s|%d|%d", merchantID, windowStart.Unix(), windowEnd.Unix())))
	return hex.EncodeToString(h[:])
}
```

### 4.2 `queue/kafka.go` (기존 스텁 대체)

```go
package queue

import (
	"context"
	"encoding/json"
	"errors"
	"fmt"
	"log/slog"
	"time"

	kafka "github.com/segmentio/kafka-go" // 가정: go.mod에 아직 없음. 5절 참고 — 팀 확인 필요
)

// Settlement jobs are produced to the Kafka topic "settlement.v1".
// RabbitMQ was fully decommissioned in June 2026 — do not target it.
const (
	SettlementTopic    = "settlement.v1"
	SettlementDLQTopic = "settlement.v1.dlq"
	maxPublishAttempts = 5
)

type Publisher struct {
	w   *kafka.Writer
	dlq *kafka.Writer
}

func NewPublisher(brokers []string) *Publisher {
	return &Publisher{
		w: &kafka.Writer{
			Addr:         kafka.TCP(brokers...),
			Topic:        SettlementTopic,
			Balancer:     &kafka.Hash{}, // key(merchant 기반)로 파티셔닝 -> merchant별 순서 보존
			RequiredAcks: kafka.RequireAll,
			MaxAttempts:  1, // writer 내장 재시도 대신 아래서 백오프+DLQ를 직접 제어
		},
		dlq: &kafka.Writer{
			Addr:         kafka.TCP(brokers...),
			Topic:        SettlementDLQTopic,
			RequiredAcks: kafka.RequireAll,
		},
	}
}

// PublishSettlement publishes one settlement batch with bounded retry + backoff.
// On exhaustion it routes the job to the DLQ topic instead of silently dropping it,
// addressing the "settlement cron has no retry logic" risk.
func (p *Publisher) PublishSettlement(ctx context.Context, job SettlementJob) error {
	payload, err := json.Marshal(job)
	if err != nil {
		return fmt.Errorf("marshal settlement job: %w", err)
	}

	msg := kafka.Message{
		Key:   []byte(job.IdempotencyKey), // dedupe/ordering key — merchant/기간 파생값, PAN 아님
		Value: payload,
	}

	var lastErr error
	backoff := 200 * time.Millisecond
	for attempt := 1; attempt <= maxPublishAttempts; attempt++ {
		job.Attempt = attempt
		if err := p.w.WriteMessages(ctx, msg); err != nil {
			lastErr = err
			slog.Warn("settlement publish failed, retrying",
				"idempotency_key", job.IdempotencyKey,
				"merchant_id", job.MerchantID,
				"attempt", attempt,
				"error", err,
			)
			select {
			case <-ctx.Done():
				return ctx.Err()
			case <-time.After(backoff):
			}
			backoff *= 2
			continue
		}
		return nil
	}

	slog.Error("settlement publish exhausted retries, routing to DLQ",
		"idempotency_key", job.IdempotencyKey,
		"merchant_id", job.MerchantID,
		"last_error", lastErr,
	)
	if dlqErr := p.dlq.WriteMessages(ctx, msg); dlqErr != nil {
		return errors.Join(
			fmt.Errorf("dlq publish failed after %d attempts: %w", maxPublishAttempts, lastErr),
			dlqErr,
		)
	}
	return fmt.Errorf("settlement job %s routed to DLQ after %d attempts: %w",
		job.IdempotencyKey, maxPublishAttempts, lastErr)
}
```

### 4.3 `store/settlement_repo.go` (신규, 스키마는 가정)

```go
package store

import (
	"context"
	"database/sql" // TODO(store/db.go): sqlc 마이그레이션 이후 생성 코드로 교체
	"time"
)

// SettlementJobsRepo tracks which settlement batches have already been claimed/published,
// so a cron re-run or a DLQ replay cannot cause a duplicate Kafka publish.
//
// 주의: 테이블/컬럼명은 스케치입니다. store/db.go에는 실제 스키마가 전혀 없어서
// (TODO 주석 한 줄뿐) 검증되지 않은 가정입니다. DB 엔진도 불명이라 아래는
// Postgres 문법($1 플레이스홀더, ON CONFLICT)으로 썼습니다 — MySQL 등이면 조정 필요.
type SettlementJobsRepo struct {
	db *sql.DB
}

func NewSettlementJobsRepo(db *sql.DB) *SettlementJobsRepo {
	return &SettlementJobsRepo{db: db}
}

// TryClaim inserts a dedupe row iff it doesn't already exist. If it returns
// claimed=false, another run already owns this batch — the caller must skip
// publishing rather than send a duplicate.
func (r *SettlementJobsRepo) TryClaim(ctx context.Context, idempotencyKey, merchantID string) (claimed bool, err error) {
	res, err := r.db.ExecContext(ctx, `
		INSERT INTO settlement_job_dedupe (idempotency_key, merchant_id, status, created_at)
		VALUES ($1, $2, 'publishing', $3)
		ON CONFLICT (idempotency_key) DO NOTHING
	`, idempotencyKey, merchantID, time.Now().UTC())
	if err != nil {
		return false, err
	}
	n, err := res.RowsAffected()
	if err != nil {
		return false, err
	}
	return n == 1, nil
}

func (r *SettlementJobsRepo) MarkPublished(ctx context.Context, idempotencyKey string) error {
	_, err := r.db.ExecContext(ctx, `
		UPDATE settlement_job_dedupe SET status = 'published', published_at = $2
		WHERE idempotency_key = $1
	`, idempotencyKey, time.Now().UTC())
	return err
}

// MarkFailed lets a claimed-but-never-published row be released so a later
// cron tick can retry it, instead of it being stuck "publishing" forever.
func (r *SettlementJobsRepo) MarkFailed(ctx context.Context, idempotencyKey string) error {
	_, err := r.db.ExecContext(ctx, `
		UPDATE settlement_job_dedupe SET status = 'failed' WHERE idempotency_key = $1
	`, idempotencyKey)
	return err
}
```

### 4.4 `main.go` (크론 결선)

```go
package main

import (
	"context"
	"log/slog"
	"os"
	"os/signal"
	"syscall"
	"time"

	"github.com/robfig/cron/v3" // 가정: go.mod에 아직 없음. 5절 참고

	"payments/queue"
	"payments/store"
)

func main() {
	ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
	defer stop()

	db := store.MustOpen(os.Getenv("DATABASE_URL")) // store/db.go에 아직 없음 — 스케치용 시그니처
	repo := store.NewSettlementJobsRepo(db)
	pub := queue.NewPublisher([]string{os.Getenv("KAFKA_BROKERS")}) // 리전별/공유 클러스터 여부 불확실(5절)

	c := cron.New()
	if _, err := c.AddFunc("@every 5m", func() {
		runCtx, cancel := context.WithTimeout(ctx, 4*time.Minute)
		defer cancel()
		if err := runSettlementBatch(runCtx, repo, pub); err != nil {
			slog.Error("settlement batch run failed", "error", err)
		}
	}); err != nil {
		slog.Error("failed to schedule settlement cron", "error", err)
		os.Exit(1)
	}
	c.Start()
	defer c.Stop()

	<-ctx.Done()
	slog.Info("shutting down")
}

func runSettlementBatch(ctx context.Context, repo *store.SettlementJobsRepo, pub *queue.Publisher) error {
	windowEnd := time.Now().UTC().Truncate(5 * time.Minute)
	windowStart := windowEnd.Add(-5 * time.Minute)

	merchants, err := repo.PendingMerchants(ctx, windowStart, windowEnd) // 스케치, 미정의
	if err != nil {
		return err
	}

	for _, m := range merchants {
		key := queue.NewIdempotencyKey(m.ID, windowStart, windowEnd)

		claimed, err := repo.TryClaim(ctx, key, m.ID)
		if err != nil {
			slog.Error("claim failed", "merchant_id", m.ID, "error", err)
			continue
		}
		if !claimed {
			continue // 다른 실행이 이미 이 배치를 소유 — 재발행 금지
		}

		job := queue.SettlementJob{
			IdempotencyKey: key,
			MerchantID:     m.ID,
			WindowStart:    windowStart,
			WindowEnd:      windowEnd,
			AmountMinor:    m.PendingAmountMinor,
			Currency:       m.Currency,
			CreatedAt:      time.Now().UTC(),
		}

		if err := pub.PublishSettlement(ctx, job); err != nil {
			slog.Error("publish failed even after DLQ routing", "merchant_id", m.ID, "error", err)
			_ = repo.MarkFailed(ctx, key) // 다음 크론 틱에서 재시도 가능하도록 해제
			continue
		}
		if err := repo.MarkPublished(ctx, key); err != nil {
			slog.Error("mark published failed", "merchant_id", m.ID, "error", err)
		}
	}
	return nil
}
```

### 4.5 Consumer (범위 불확실 — 참고용 최소 스켈레톤)

이 저장소에는 producer 스텁만 있고 `settlement.v1`을 소비하는 코드가 없습니다. 별도 서비스가 소비하는 구조일 수도 있어 **이 부분은 이 repo 소관인지부터 팀 확인이 필요**합니다. 만약 이 repo가 소비까지 맡는다면, 최소한 아래 형태로 idempotent upsert + DLQ 재처리 구조를 맞춰야 큐잉 개선이 끝까지 의미를 가집니다.

```go
package queue

import (
	"context"
	kafka "github.com/segmentio/kafka-go"
)

// Consumer sketch only — no existing consumer code was found in this repo to build on.
type Consumer struct {
	r *kafka.Reader
}

func NewConsumer(brokers []string, groupID string) *Consumer {
	return &Consumer{r: kafka.NewReader(kafka.ReaderConfig{
		Brokers: brokers,
		GroupID: groupID,
		Topic:   SettlementTopic,
	})}
}

func (c *Consumer) Run(ctx context.Context, handle func(context.Context, SettlementJob) error) error {
	for {
		m, err := c.r.FetchMessage(ctx)
		if err != nil {
			return err
		}
		var job SettlementJob
		if err := json.Unmarshal(m.Value, &job); err != nil {
			// 역직렬화 실패 — 재시도 무의미, DLQ로 넘기고 커밋
			_ = c.r.CommitMessages(ctx, m)
			continue
		}
		// handle()은 store 쪽에서 idempotency_key로 upsert(멱등)해야 함 —
		// 컨슈머 재시작/리밸런스로 인한 재전달(at-least-once)을 전제로 설계.
		if err := handle(ctx, job); err != nil {
			continue // 커밋 보류 -> 재전달됨. 별도 재시도 횟수 카운트 후 DLQ 라우팅 필요(스케치 생략)
		}
		_ = c.r.CommitMessages(ctx, m)
	}
}
```

## 5. 명시적 가정/불확실 항목 (팀 확인 필요)

- **Kafka 클라이언트 라이브러리 미확정.** `go.mod`에 의존성이 전혀 없어서 `segmentio/kafka-go`로 스케치했습니다. 브로커 레벨 idempotent producer(`enable.idempotence`, sequence number 등)가 꼭 필요하면 `IBM/sarama`(`config.Producer.Idempotent = true`)가 더 성숙한 지원을 제공하므로 대안으로 남겨둡니다. 이 스케치는 그 대신 애플리케이션 레벨(DB claim) idempotency를 1차 방어선으로 삼아 클라이언트 라이브러리 선택에 덜 의존하게 했습니다.
- **Kafka 클러스터 토폴로지 불명.** `deploy/regions.yaml`은 3리전을 나열할 뿐, Kafka가 리전별 독립 클러스터인지 리전 간 복제(MirrorMaker2 등)가 있는지는 코드에 없습니다. 리전당 별도 클러스터라면 파티션 키/컨슈머 그룹 설계가 달라질 수 있어 확인 전에는 확정하지 않았습니다.
- **DB 엔진/스키마 불명.** `store/db.go`가 빈 스텁이라 실제 테이블 구조와 엔진(Postgres/MySQL 등)을 알 수 없습니다. 4.3의 SQL은 Postgres 문법 가정이며 실제 스키마와 다를 수 있습니다.
- **크론 스케줄러 라이브러리 미확정.** `robfig/cron/v3`로 스케치했으나 팀이 이미 다른 방식(k8s CronJob, systemd timer 등)을 쓰기로 했을 수 있어 확정된 선택이 아닙니다.
- **정산 크론이 리전마다 독립 실행되는지, 단일 리더가 전역 발행하는지 불명.** 멀티 리전 배포(3곳)에서 각 리전 인스턴스가 동시에 크론을 돌리면 같은 배치를 여러 리전이 동시에 claim 시도할 수 있어 DB claim이 유일한 방어선이 됩니다. 리전 간 조정(리더 선출 등)이 필요한지는 인프라팀 확인이 필요합니다.
- **consumer 소재 불명.** 위 4.5에서 언급한 대로, 이 repo가 소비까지 담당하는지 확인되지 않았습니다.
- **charge.go의 idempotency FIXME는 여기서 해결되지 않습니다.** 정산 잡 dedupe와는 별개 문제이며 별도 조치가 필요합니다.

## 6. 요약

- 아티팩트의 "RabbitMQ / 단일 리전" 결정 슬롯은 코드(Kafka 전환, 3리전 배포)와 모순되는 stale 정보로 판단해 채택하지 않았고, Kafka + 멀티 리전 전제로 스케치했습니다.
- 핵심 개선안은 (1) merchant×시간창 기반 결정적 idempotency key로 DB claim 후에만 발행, (2) producer 지수 백오프 재시도 + `settlement.v1.dlq`, (3) `main.go` 크론 결선이며, Kafka 클라이언트/DB 엔진/클러스터 토폴로지/consumer 소재는 저장소에 근거가 없어 확정하지 않고 가정으로 명시했습니다.
