## 1. 중요: 아티팩트의 "Known Decisions"는 일부 stale — 이 계획은 코드를 우선함

`artifact-M9.md`의 Known Decisions 슬롯은 다음을 주장한다.

> Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there.
> Payments runs single-region in us-east-1; DR is cold-standby only.

그런데 실제 코드는 정반대다.

- `deploy/regions.yaml` (주석: "active deployment topology (July 2026)")에 `us-east-1`, `eu-west-1`, `ap-northeast-2` **3개 리전**이 명시돼 있다. → 단일 리전 아님, **멀티 리전**.
- `queue/kafka.go`: "Settlement jobs are produced to the Kafka topic `settlement.v1`. RabbitMQ was fully decommissioned in June 2026 — do not target it." → RabbitMQ는 이미 폐기됐고 현재는 **Kafka**(`settlement.v1`)를 쓴다.
- `README.md`도 "Settlement jobs flow through Kafka (queue/); deployed multi-region (deploy/regions.yaml)"라고 코드와 동일하게 서술.
- git 로그상 가장 최근 커밋(`74d4646`, "kafka settlement + multi-region topology")이 바로 이 전환을 기록하고 있고, 그 이전 커밋들에는 리전/큐 관련 언급이 전혀 없다. 즉 이 변경이 최신 상태이고, 아티팩트가 그걸 못 따라간 것으로 보인다.

**판단**: 아티팩트의 이 두 항목은 outdated이며, 그대로 따르면 이미 없는 RabbitMQ를 새 컨슈머 바인딩 대상으로 안내하는 등 실제로 위험하다. 이 계획은 코드 근거(3-리전 + Kafka)를 채택한다. `artifact-M9.md`는 Known Decisions 슬롯 재검증/재컴파일이 필요해 보이지만 그 작업은 이 문서의 범위 밖이다.

아티팩트의 나머지 Known Decisions — PAN은 vault 서비스에만 저장 / JWT 15분 만료 인증 / 환불은 provider webhook 비동기 처리 — 는 repo 안에 관련 코드가 전혀 없어(vault/auth/webhook 키워드 grep 결과 0건) 코드로 확인도 반박도 안 된다. 반박 근거가 없으니 잠정 채택하되, "코드로 검증됨"이 아니라 **"미검증 주장"**으로 구분해서 아래에 표시한다.

## 4. 장애 시나리오별 대응 방향 (초안 — 세부는 3장 항목 확정 후 구체화 필요)

### 4.1 리전 1곳 완전 장애 (예: eu-west-1 다운)

1. 감지: 해당 리전 헬스체크/에러율 급증 — 알림 경로는 [불명/TBD], 확인 즉시 이 문서에 채워야 함.
2. 트리아지: 영향 범위를 "결제(charge)", "정산(settlement)", "환불(refund)" 축으로 분류. 리전 장애가 곧 전체 서비스 장애는 아닐 수 있음(멀티 리전이므로) — 단, 라우팅 메커니즘 미확인이라 "자동으로 다른 리전이 받아준다"고 가정하지 말고 실측 확인.
3. 트래픽 격리: 장애 리전으로의 신규 라우팅 차단. **구체적 콘솔/명령 절차는 실제 라우팅 계층(3장 참고) 확인 전에는 채울 수 없음.**
4. Kafka 영향 확인: 장애 리전에 물려 있던 프로듀서/컨슈머의 처리 상태 점검. 리전 간 복제 여부가 불명확하므로, 장애 리전에서 진행 중이던 정산 이벤트의 유실 가능성을 "없다"고 단정하지 말 것.
5. 잔존 리전 capacity 확인: 나머지 2개 리전이 이관된 트래픽을 감당하는지 실시간 모니터링.
6. 페일백: 리전 복구 후 트래픽을 점진적으로 되돌리고, 정산 큐 상태(누락/중복) 재동기화 확인.

### 4.2 Kafka(`settlement.v1`) 장애 또는 심각한 지연

- 정산 배치 지연/유실 위험. 단, `main.go`의 settlement cron이 이 저장소 기준으로는 아직 연결조차 안 된 상태(TODO)라, **실제 프로덕션에서 크론이 이미 구현/배포돼 있는지부터 먼저 확인**해야 한다. 이 저장소의 스텁 상태를 프로덕션 상태와 동일시하면 안 된다.
- idempotency key가 강제되지 않는 상태(`handlers/charge.go` FIXME)이므로, 장애 후 큐 재처리·재시도 과정에서 **동일 결제가 중복 처리될 위험**이 실질적으로 존재한다. 재시도를 자동으로 트리거하기 전에 반드시 수동 중복 검증 게이트를 둔다.
- 아티팩트 Known Risks에도 "정산 크론에 재시도 로직이 없어 보인다"는 지적이 있다 — 이 역시 재확인 대상.

### 4.3 DB/store 계층 장애

- `store/db.go`가 빈 스텁이라 실제 DB 엔진, 복제 구성, 백업 주기(=RPO의 실질적 하한)를 이 저장소에서 알 수 없다. 이 시나리오는 실제 DB 인프라 문서를 확보하기 전에는 절차를 채울 수 없는 영역으로 남긴다.

### 4.4 Vault(카드 PAN 조회) 서비스 리전 장애

- vault가 API 레이어와 분리된 별도 서비스라는 것은 아티팩트 주장일 뿐 코드 근거가 없다. vault 이중화 여부, 장애 시 charge/refund 흐름이 어떻게 저하되는지(하드 실패 vs 큐잉 후 재시도) 모두 [불명/TBD]. 대응 중 임시 디버깅이 필요해도 `CLAUDE.md`의 "Never log PANs" 규칙은 예외 없이 지킨다.

## 9. 요약

- 코드 기준 현재 토폴로지는 **3-리전(us-east-1, eu-west-1, ap-northeast-2) 멀티 리전 + Kafka(`settlement.v1`) 정산 파이프라인**이다. 아티팩트에 남아 있는 "단일 리전 + RabbitMQ" 서술은 outdated이므로 이 계획에서는 배제했다.
- 실제 라우팅/DB/Kafka 클러스터 구성/모니터링 세부는 이 저장소에 없어 다수 항목을 TBD로 남겼다. 임의 가정을 넣지 않았으므로, 이 계획은 "실행 가능한 런북"이 아니라 그 전 단계의 뼈대/체크리스트로 취급해야 한다.
- 가장 시급한 실행 리스크는 charge 핸들러의 idempotency key 미강제이며, 리전 페일오버 시 재시도가 겹치면 중복 결제로 직결되므로 해결 전까지는 자동 재시도 대신 수동 검증 게이트를 두어야 한다.
