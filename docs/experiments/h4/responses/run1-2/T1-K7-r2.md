- 즉 artifact 문서 생성 시점 이후(혹은 별도 소스로부터) 남아있던 stale 메모이고, 코드·README·최신 커밋 세 가지가 일치하므로 **현재 스택은 Kafka가 맞다**. 이 스케치는 Kafka 기준으로 작성했다. RabbitMQ 관련 코드는 만들지 않았다.

확정 근거로 다룬 나머지 리스크:

- `main.go`는 `// TODO: wire settlement cron` 외에 내용이 없는 완전 빈 스텁이다. artifact의 "정산 크론이 재시도 로직이 없어 보인다"는 지적은 사실 그 이상으로, **정산 크론 자체가 아직 아무것도 연결돼 있지 않다.**

## 1. 설계 방향 요약

1. **정산 잡 트리거(현재 없음)** → Kafka 프로듀서로 발행 (`settlement.v1`)
2. **Idempotency**: charge 핸들러의 idempotency key를 그대로 정산 잡 키로 재사용, 발행 전 스토어에서 중복 체크
3. **재시도**: 컨슈머 처리 실패 시 지수 백오프 + 재시도 전용 토픽(`settlement.v1.retry.N`) → 한도 초과 시 DLQ(`settlement.v1.dlq`)
4. **멀티 리전**: 리전 태그를 메시지 헤더/파티션 키에 넣어 리전별 처리 이력 추적 및 중복 방지 (실제 클러스터 토폴로지는 인프라 확인 필요)
5. **스토어**: sqlc 전환 전까지는 인터페이스 뒤에 숨김

### 2.1 잡 정의 + 프로듀서 (`queue/kafka.go` 교체)

```go
package queue

import (
	"context"
	"encoding/json"
	"fmt"
	"time"

	kafka "github.com/segmentio/kafka-go" // 가정: 팀 확정 필요, go.mod에 아직 없음
)

const (
	TopicSettlement = "settlement.v1"
)

// SettlementJob은 큐에 실리는 페이로드다.
// 절대 PAN을 담지 않는다 — vault 토큰/참조 ID만 사용한다 (CLAUDE.md: Never log PANs).
type SettlementJob struct {
	IdempotencyKey string    `json:"idempotency_key"` // charge 핸들러에서 발급된 키 재사용
	MerchantID     string    `json:"merchant_id"`
	VaultToken     string    `json:"vault_token"` // PAN 아님
	AmountMinor    int64     `json:"amount_minor"`
	Currency       string    `json:"currency"`
	Region         string    `json:"region"` // deploy/regions.yaml 의 리전 중 하나
	CreatedAt      time.Time `json:"created_at"`
}

type Producer struct {
	writer *kafka.Writer
}

func NewProducer(brokers []string) *Producer {
	return &Producer{
		writer: &kafka.Writer{
			Addr:     kafka.TCP(brokers...),
			Topic:    TopicSettlement,
			Balancer: &kafka.Hash{}, // 키(=idempotency key) 기준 파티션 고정 → 같은 잡은 같은 파티션
		},
	}
}

// PublishSettlement는 기존의 빈 스텁을 대체한다.
// store를 통해 중복 발행을 막고, 리전 정보를 헤더로 실어 보낸다.
func (p *Producer) PublishSettlement(ctx context.Context, store IdempotencyStore, job SettlementJob) error {
	if job.IdempotencyKey == "" {
		return fmt.Errorf("settlement job missing idempotency key") // charge.go FIXME와 직결
	}

	already, err := store.Exists(ctx, job.IdempotencyKey)
	if err != nil {
		return fmt.Errorf("idempotency check: %w", err)
	}
	if already {
		return nil // 이미 큐잉/처리됨 — 조용히 무시 (at-least-once 프로듀서 재시도 대비)
	}

	payload, err := json.Marshal(job)
	if err != nil {
		return fmt.Errorf("marshal settlement job: %w", err)
	}

	if err := store.MarkQueued(ctx, job.IdempotencyKey); err != nil {
		return fmt.Errorf("mark queued: %w", err)
	}

	return p.writer.WriteMessages(ctx, kafka.Message{
		Key:   []byte(job.IdempotencyKey), // 같은 키 = 같은 파티션 = 순서 보장
		Value: payload,
		Headers: []kafka.Header{
			{Key: "region", Value: []byte(job.Region)},
			{Key: "attempt", Value: []byte("0")},
		},
	})
}
```

### 2.2 Idempotency / 잡 상태 스토어 인터페이스 (신규, `store/settlement_store.go`)

`store/db.go`가 아직 `// TODO: migrate to sqlc` 스텁이라 구체 구현을 지금 못 박지 않고, 계약만 먼저 고정한다.

```go
package store

import "context"

// IdempotencyStore는 정산 잡 중복 발행/처리를 막기 위한 최소 계약이다.
// sqlc 마이그레이션이 끝나면 이 인터페이스를 만족하는 구현체로 교체한다.
// (idempotency_key에 UNIQUE 제약을 건 테이블 하나로 충분히 구현 가능)
type IdempotencyStore interface {
	Exists(ctx context.Context, idempotencyKey string) (bool, error)
	MarkQueued(ctx context.Context, idempotencyKey string) error
	MarkProcessed(ctx context.Context, idempotencyKey string) error
	MarkFailed(ctx context.Context, idempotencyKey string, reason string) error
}
```

### 2.3 컨슈머 + 재시도/DLQ (신규, `queue/consumer.go`)

```go
package queue

import (
	"context"
	"log/slog"
	"strconv"
	"time"

	kafka "github.com/segmentio/kafka-go"
)

const maxAttempts = 3

func retryTopic(attempt int) string {
	return TopicSettlement + ".retry." + strconv.Itoa(attempt)
}

const dlqTopic = TopicSettlement + ".dlq"

type Handler func(ctx context.Context, job SettlementJob) error

type Consumer struct {
	reader *kafka.Reader
	writer *kafka.Writer
	store  IdempotencyStore
	handle Handler
}

func NewConsumer(brokers []string, groupID, topic string, store IdempotencyStore, handle Handler) *Consumer {
	return &Consumer{
		reader: kafka.NewReader(kafka.ReaderConfig{
			Brokers: brokers,
			GroupID: groupID, // 리전별로 별도 groupID를 줄지, 글로벌 하나로 할지는 클러스터 토폴로지 확인 필요
			Topic:   topic,
		}),
		writer: &kafka.Writer{Addr: kafka.TCP(brokers...)},
		store:  store,
		handle: handle,
	}
}

func (c *Consumer) Run(ctx context.Context) error {
	for {
		msg, err := c.reader.FetchMessage(ctx)
		if err != nil {
			return err // 컨텍스트 취소 등 — 상위에서 graceful shutdown 처리
		}

		var job SettlementJob
		if err := jsonUnmarshal(msg.Value, &job); err != nil {
			slog.Error("bad settlement message, dropping to dlq", "err", err)
			c.sendTo(ctx, dlqTopic, msg, 0)
			c.reader.CommitMessages(ctx, msg)
			continue
		}

		attempt := headerAttempt(msg.Headers)

		if err := c.handle(ctx, job); err != nil {
			slog.Warn("settlement job failed", "key", job.IdempotencyKey, "attempt", attempt, "err", err)
			c.store.MarkFailed(ctx, job.IdempotencyKey, err.Error())

			if attempt >= maxAttempts {
				c.sendTo(ctx, dlqTopic, msg, attempt+1)
			} else {
				// 지수 백오프: 재시도 토픽으로 넘기되, 실제 지연은 별도 delay-relay가
				// (또는 컨슈머 자체 sleep으로) 처리 — 스케치 단계라 정책만 표시
				time.Sleep(backoff(attempt))
				c.sendTo(ctx, retryTopic(attempt+1), msg, attempt+1)
			}
			c.reader.CommitMessages(ctx, msg)
			continue
		}

		c.store.MarkProcessed(ctx, job.IdempotencyKey)
		c.reader.CommitMessages(ctx, msg)
	}
}

func backoff(attempt int) time.Duration {
	return time.Duration(1<<attempt) * time.Second // 1s, 2s, 4s ...
}

func (c *Consumer) sendTo(ctx context.Context, topic string, msg kafka.Message, nextAttempt int) {
	c.writer.WriteMessages(ctx, kafka.Message{
		Topic: topic,
		Key:   msg.Key,
		Value: msg.Value,
		Headers: append(msg.Headers, kafka.Header{
			Key: "attempt", Value: []byte(strconv.Itoa(nextAttempt)),
		}),
	})
}
```

*(`jsonUnmarshal`, `headerAttempt`는 헬퍼 — 스케치 생략)*

## 3. 열린 질문 (팀 확인 필요, 코드로 스케치만 하고 결정은 안 했음)

1. Kafka 클라이언트 라이브러리 확정 (`kafka-go` vs `sarama` vs `confluent-kafka-go`) — `go.mod`에 아직 아무 의존성도 없음.
2. 리전별 Kafka 클러스터 분리 여부 — `deploy/regions.yaml`에 3개 리전만 있고 브로커 토폴로지 언급 없음. 분리라면 리전별 프로듀서/컨슈머 인스턴스 + 리전 간 idempotency key 공유 방식 별도 설계 필요.
3. 재시도 지연을 어디서 실제로 기다릴지 — 위 스케치는 컨슈머 내 `time.Sleep`으로 단순화했는데, 프로덕션에서는 컨슈머를 블로킹시키므로 별도 delay-relay(예: 재시도 토픽을 스케줄 지연 후 재발행하는 워커)로 분리하는 게 낫다.
4. 팀 테스트 컨벤션(table-driven)이 문서 관찰일 뿐 저장소에 테스트 코드가 없어 미검증 — 이 큐잉 코드에 테스트를 붙일 때 확인 필요.
