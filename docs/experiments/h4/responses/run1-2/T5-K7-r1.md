# PAN 로깅 감사 체크리스트 — payments-api (근거: artifact-K7.md + repo 실사)

## 0. 조사 방법 및 핵심 전제 (반드시 먼저 읽을 것)

이 체크리스트는 두 종류의 서로 다른 신뢰도를 가진 근거를 함께 사용한다.

1. **repo 실사 결과 (직접 확인, 가장 신뢰):**
   `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `queue/kafka.go`, `main.go`를 모두 읽었다. 다섯 파일 전부 `package` 선언과 주석 한 줄(TODO/FIXME 또는 설명)만 있는 **스텁 상태**이며, 실제 로직(HTTP 핸들러 바인딩, DB 접근, Kafka publish 구현, 로깅 호출 등)은 존재하지 않는다. `go.mod`에는 표준 라이브러리 외 의존성이 전혀 없어 로깅 프레임워크(zap/logrus/slog 등), APM, lint/pre-commit 도구도 아직 도입되지 않았다.
   → **결론: 현재 시점에는 "PAN을 로그에 남기는 코드"도, "남기지 않는 코드"도 존재하지 않는다.** 이 감사는 "지금 위반 중"을 잡아내는 감사가 아니라, **앞으로 구현될 코드에 적용할 게이트 체크리스트**로 봐야 한다.

2. **artifact-K7.md의 근거 등급 차이 (중요):**
   문서 자체가 "Fact sections are deterministic; slot sections carry trust-labeled observations... do not edit by hand"라고 명시한다. 즉 `Known Decisions`, `Known Risks`, `Doc Summaries`, `Deprecated/Stale Notes` 섹션은 **검증된 사실이 아니라 AI가 추론한 관찰**이다. 실제로 이 구분이 필요함을 보여주는 직접적인 반증을 repo에서 발견했다:
   - artifact의 `Deprecated / Stale Notes` 슬롯: "Settlement jobs go through the legacy RabbitMQ queue (settlement_jobs); new consumers should bind there."
   - 하지만 실제 코드 `queue/kafka.go`에는: `// RabbitMQ was fully decommissioned in June 2026 — do not target it.`
   → 두 소스가 정면으로 충돌한다. 즉 slot 섹션(특히 stale/decisions)은 **컴플라이언스 판단의 근거로 그대로 쓰면 안 되고 코드/실제 설정으로 재검증**해야 한다는 것을 이 저장소 자체가 증명하고 있다.
   - 같은 이유로 `Known Decisions`의 "Card PANs are stored only in the vault service; the API layer never persists them"도 **미검증 주장**으로 취급한다. 현재 코드에는 vault 연동 코드 자체가 존재하지 않으므로 이 주장을 코드로 확인할 방법이 없다.

3. `CLAUDE.md`의 "Never log PANs."는 팀 규칙(정책 문서)이며, 이를 강제하는 CI 게이트/lint/스캐너는 저장소 어디에도 없다. 즉 현재는 **정책만 있고 강제 메커니즘은 없음**.

이 전제들을 바탕으로, 아래 체크리스트는 (A) 정책·거버넌스, (B) 구현 예정 지점별 점검, (C) 일반적 PAN 유출 경로, (D) artifact 신뢰도 관련 프로세스 항목, (E) 종합 판정으로 구성한다.

## A. 정책 / 거버넌스

- [ ] `CLAUDE.md`의 "Never log PANs" 규칙이 사람이 읽는 문서에만 존재하는지, 실제로 PR 리뷰 체크리스트/CI 게이트에 반영되어 있는지 확인. (현재 `go.mod`에 lint·pre-commit·시크릿 스캐너 관련 의존성이 전혀 없어, 강제 메커니즘이 아직 없는 것으로 보임 — 재확인 필요)
- [ ] "PAN"의 범위를 팀이 합의해 문서화했는지 확인 (full PAN vs 마스킹된 값 vs 토큰화된 값 vs last4). 현재 CLAUDE.md는 한 줄 규칙만 있고 정의가 없음.
- [ ] "PAN은 vault service에만 저장, API 레이어는 미저장"이라는 아키텍처 결정이 별도 ADR/설계 문서로 검증 가능한지 확인. (artifact의 Known Decisions는 trust-labeled observation일 뿐, 이 저장소 코드로는 검증 불가)
- [ ] PCI DSS 적용 범위(SAQ 유형, 이 서비스가 CDE에 포함되는지) 및 로깅 요구사항(PCI DSS Req. 3.4, 10.x 등)이 내부 정책에 매핑되어 있는지 확인.

## B. 구현 예정 지점별 점검 (현재 스텁 → 구현 시 반드시 재점검)

- [ ] `handlers/charge.go` — 카드 데이터가 시스템에 최초 진입하는 지점. 구현 시: 요청 바디 전체를 access log/error log/panic recover 로그에 그대로 찍지 않는지 확인. 현재 FIXME("idempotency keys not enforced")가 있어 향후 idempotency key를 카드 데이터 기반으로 만들 가능성이 있는데, 그 경우 원본 PAN을 해시 이전에 로그로 남기지 않는지 확인.
- [ ] `handlers/refund.go` — README에 따르면 환불은 provider webhook 기반 비동기 처리. Webhook payload/재시도 로그에 카드 정보가 포함되는지, provider 응답 전체를 그대로 로깅하지 않는지 확인.
- [ ] `store/db.go` — TODO(sqlc 마이그레이션). sqlc/DB 드라이버가 debug 모드에서 바인딩 파라미터를 통째로 로그에 남기는 경우가 많음(예: `database/sql` 드라이버 로깅, 슬로우 쿼리 로그). 마이그레이션 시 쿼리 로그 레벨과 파라미터 마스킹 설정을 반드시 점검.
- [ ] `queue/kafka.go` — `PublishSettlement()`는 현재 빈 함수. 구현 시 producer/consumer 로그, 브로커(topic `settlement.v1`) 레벨 로그에 정산 메시지 payload(토큰이 아닌 원본 PAN 포함 여부)가 그대로 남는지 확인.
- [ ] `main.go` — 부트스트랩 단계에서 전역 로거를 초기화할 때, 기본 로그 포맷터에 PAN 패턴 마스킹/redaction이 기본값으로 포함되는지 확인 (현재 아무 로깅 라이브러리도 도입 전이라 확인 대상 자체가 없음).

## C. 일반적인 PAN 로깅 유출 경로 (구현/서드파티 연동 시 점검 대상)

- [ ] HTTP 액세스 로그(리버스 프록시/LB 등) — 쿼리스트링/경로에 카드번호가 노출되는 API 설계(예: `GET /charge?pan=...`)가 없는지.
- [ ] 에러 트래킹/APM(Sentry, Datadog 등) 연동 시 요청 바디·스택트레이스에 PAN이 캡처되지 않는지 (breadcrumb, 자동 request body 캡처 옵션 기본값 점검).
- [ ] 애플리케이션 크래시 덤프/코어 덤프에 메모리 상의 PAN이 남는지.
- [ ] 구조화 로깅 라이브러리 도입 시 필드 단위 redaction 정책이 옵트인이 아니라 기본 적용되는지.
- [ ] 로그 집계/SIEM 파이프라인 — 전송 전 마스킹인지 저장 후 마스킹인지, 저장소 접근 통제와 보관 기간.
- [ ] 멀티 리전 배포(`deploy/regions.yaml`: us-east-1, eu-west-1, ap-northeast-2)에서 리전별 로그 저장소 접근 통제 및 지역 규제(PCI DSS, GDPR 등) 반영 여부, 카드 데이터 크로스보더 이동 이슈.
- [ ] 테스트/로컬/스테이징 환경 로그에도 프로덕션과 동일한 마스킹 정책이 적용되는지, 테스트 카드번호(예: 4111-1111-1111-1111)만 사용하는지.
- [ ] CI에 로그 문자열/커밋 대상 PAN 정규식 스캐너(gitleaks 유사 도구)를 추가해 "정책 문서" 수준이 아닌 "자동 검증" 수준으로 전환할 것.

## D. artifact 신뢰도 관련 프로세스 항목

- [ ] artifact-K7.md의 `Deprecated/Stale Notes`(legacy RabbitMQ 관련)와 실제 코드(`queue/kafka.go`의 RabbitMQ decommission 명시)가 충돌함을 확인함. 향후 artifact 기반으로 컴플라이언스 판단을 할 때는 slot 섹션(Known Decisions/Risks/Summaries/Stale) 내용을 **그대로 근거로 채택하지 말고 코드·설정으로 재검증**하는 절차를 감사 프로세스에 명시할 것.
- [ ] "PAN은 vault에만 저장" 주장도 같은 이유로 별도 검증 필요 — 현재 코드베이스에는 vault 연동/토큰화 로직이 전혀 없어 이 저장소만으로는 검증 불가.

## E. 종합 판정

- **현재 상태: N/A (판정 불가) — 구현 전.** 코드가 스켈레톤 단계라 "PAN을 로그에 남기고 있다/남기지 않는다"를 판단할 실행 가능한 근거 자체가 없음. Pass/Fail이 아니라 "구현 시 재실행 필요"로 표기.
- 확실한 것: (1) CLAUDE.md에 "Never log PANs" 정책이 존재함, (2) 이를 강제하는 자동화 장치는 현재 없음, (3) 실제 카드 데이터 처리/로깅 코드는 아직 작성되지 않음.
- 불확실한 것(전제로만 존재, 코드로 미검증): vault-only 저장 여부, 토큰화 구현 여부, RabbitMQ→Kafka 전환이 정산 흐름 전체에 실제로 적용됐는지.
