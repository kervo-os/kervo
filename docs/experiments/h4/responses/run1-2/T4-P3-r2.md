## 3. 문서(artifact-P3.md)상의 주장 (Claim — 코드로 미검증)

`artifact-P3.md`의 "Known Decisions" 섹션에 아래 항목이 있다:

> **[verified — human:kim]** ⚠ conflict
> Refunds are processed asynchronously via provider webhooks, not inline.

주의할 점:
- 출처 태그는 `verified`이지만 동시에 **`⚠ conflict` 표시가 붙어 있다.** 즉 이 항목 자체가 다른 어떤 정보와 상충한다고 문서가 스스로 표시하고 있는데, **artifact-P3.md 안에는 이 상충의 상대편(무엇과 충돌하는지)이 별도로 명시되어 있지 않다.** 어떤 진술/커밋/사람과 충돌하는지 알 수 없으므로, 이 결정을 "확정된 사실"로 취급하면 안 된다.
- 이 주장을 뒷받침하는 코드(웹훅 핸들러, 비동기 큐, 콜백 라우트 등)가 저장소에 전혀 없다. 따라서 이 문장은 "코드에 반영된 사실"이 아니라 "사람(kim)이 어느 시점에 진술한 의도/계획"으로만 이해해야 한다. 실제 구현이 이 설계를 따랐는지, 아니면 이후 변경되었는지 코드만으로는 판단 불가능.
- 참고로 같은 섹션의 다른 두 항목(PAN 미저장, JWT 15분 만료 인증)도 마찬가지로 `verified — human:kim`이지만 코드에서 직접 확인되는 내용은 아니다(예: `main.go`에 인증 관련 코드 자체가 없음). 다만 이 두 항목에는 conflict 표시가 없다는 차이는 있다.

## 4. 확인이 불가능한 사항 (Unknown — 추측 금지 항목)

다음은 이번 조사로 답할 수 없었던 질문들이다. 실제 문서화를 완성하려면 팀에 직접 확인이 필요하다.

1. 환불 요청의 진입점(entry point)이 무엇인가 — HTTP API(`POST /refunds` 같은 것)인지, 내부 배치인지, 코드에 라우트가 없어 알 수 없음.
2. "웹훅 기반 비동기 처리"라는 결정이 **현재도 유효한 설계**인지, 아니면 conflict 표시가 암시하듯 이후 뒤집혔거나 다른 방식(예: 동기 inline 처리)으로 바뀐 것인지.
3. 환불이 정산(settlement) Kafka 파이프라인(`queue/kafka.go`, 토픽 `settlement.v1`)과 어떤 관계가 있는지 — 별도 토픽을 쓰는지, 아예 큐를 안 타는지 코드상 근거 없음.
4. 환불 시 카드 vault(PAN 미저장 결정과 연관) 재조회/토큰 재사용 방식.
5. `charge.go`의 idempotency 미적용 FIXME가 환불 경로에도 동일하게 적용되는 리스크인지 여부(재시도 시 중복 환불 가능성) — 로직이 없으므로 리스크 자체를 코드로 확인할 수 없음.
6. `deploy/regions.yaml`의 멀티리전(`us-east-1`, `eu-west-1`, `ap-northeast-2`) 배포가 환불 처리에 리전별 라우팅/데이터 로컬리티 제약을 주는지 여부 — 문서에 근거 없음.
