## 2. 확인이 필요한 핵심 공백 (이 계획이 "초안"인 이유)

다음은 지역 장애 대응 계획에 반드시 필요하지만 저장소/아티팩트 어디에도 근거가 없는 항목이다. **실행 가능한(actionable) 런북으로 승격하기 전에 반드시 인프라팀과 확인할 것.**

1. **Active-Active vs Active-Passive 여부**: `regions.yaml`은 순서만 있는 리스트다. 세 리전이 모두 트래픽을 받는지, 하나가 primary이고 나머지가 DR인지 불명.
2. **데이터베이스 토폴로지**: `store/db.go`는 `// TODO: migrate to sqlc`라는 주석뿐, 실제 DB 엔진/리전별 복제/리더 승격(failover) 절차가 코드에 없음. 결제 원장 같은 상태 저장소가 리전 장애 시 어떻게 페일오버되는지 전혀 알 수 없음.
3. **Kafka 클러스터 구성**: `settlement.v1` 토픽이 리전별 독립 클러스터인지, 글로벌 단일 클러스터에 MirrorMaker 등으로 복제되는지 불명. 이건 "정산 잡이 리전 장애 중에도 유실 없이 처리되는가"의 핵심인데 근거가 없음.
4. **Vault 서비스의 리전 가용성**: PAN이 저장되는 vault 서비스가 어느 리전에 있는지, 다중 리전 복제가 되는지 불명. API가 vault에 접근 못 하면 charge/refund 전체가 막힐 수 있음.
5. **트래픽 라우팅/DNS/로드밸런서**: 리전 장애 시 트래픽을 어떻게 다른 리전으로 돌리는지(GSLB, Route53 헬스체크 등) 설정이 저장소에 없음.
6. **`ap-northeast-2` 포함 배경**: 데이터 레지던시(예: 한국 규제) 목적인지 단순 지연시간 목적인지 문서화 안 됨 — 리전 페일오버 시 규제 준수 제약이 있을 수 있어 확인 필요.

이 항목들을 채우지 못한 채로 실제 장애 대응에 들어가면, "리전이 죽었을 때 어디로 트래픽을 돌려야 하는가"부터 즉흥적으로 결정해야 하는 상황이 된다.

## 3. 신뢰도가 낮거나 상충하는 입력에 대한 처리

- **정산 cron 재시도 로직 부재 리스크**: artifact에 `[generated — backend:gpt-oss-120b]`로 표시되어 있음. 이는 "검증됨"이 아니라 LLM이 `main.go`의 TODO만 보고 추론한 가설이다. `main.go`를 보면 애초에 cron 자체가 배선되지 않았으므로(빈 `main()`), "재시도 로직이 없다"는 진단보다 "정산 자동화 자체가 아직 존재하지 않는다"가 더 정확한 현재 상태다. 리전 장애 대응 관점에서는 오히려 이 점이 더 중요함: **자동화된 정산 잡이 없으므로 리전 장애 중 정산 파이프라인을 "일시정지/재개"할 운영 스위치 자체가 아직 없다.**
- **환불 처리 방식의 "⚠ conflict" 플래그**: "비동기 webhook 처리"라는 결정이 verified 라벨이면서도 충돌 표시가 있다는 것은, 이 결정과 상충하는 다른 근거(다른 문서나 코드)가 어딘가 존재한다는 뜻이다. `handlers/refund.go`는 실제로는 `// refund flow`라는 주석 한 줄뿐이라 코드로도 확인이 안 된다. **이 계획에서는 환불 흐름에 대한 가정(비동기/webhook)을 "미확정"으로 취급**하고, 장애 대응 절차에서 환불 관련 단계는 최소한으로만 서술한다.
- **팀 컨벤션(테이블 기반 테스트)**: `[observed — human:lee]`로, 장애 대응 계획과 직접 관련은 없으나 향후 페일오버 로직/헬스체크를 구현할 때 테스트 스타일 참고용으로 기록.

## 4. 장애 대응 절차 (뼈대)

### 4.1 탐지 (Detection)
- 리전별 헬스체크/에러율/레이턴시 급증 알림 — **구체 임계값·대시보드는 저장소에 없음, 확인 필요**.
- Kafka `settlement.v1` 컨슈머 랙(consumer lag) 급증 — 정산 파이프라인이 리전 장애로 막혔는지의 선행 지표가 될 수 있음(단, Kafka 클러스터가 리전별인지 글로벌인지에 따라 해석이 달라짐 — §2-3 참고).

### 4.2 트리아지 (Triage)
1. 장애 리전 식별: `us-east-1` / `eu-west-1` / `ap-northeast-2` 중 어디인지.
2. 영향 범위 판단:
   - Charge API 영향 여부 (idempotency key 미강제 상태이므로, 클라이언트/게이트웨이 재시도가 있는 경우 **중복 결제 위험**이 평소보다 큼 — §5 참고)
   - Vault 접근 가능 여부 (PAN 조회/저장 불가 시 charge/refund 전면 중단 가능성)
   - Kafka 정산 토픽 발행 가능 여부
3. 영향받은 리전의 데이터가 다른 리전과 동기/비동기 복제 관계인지 확인 — **현재 근거 없음, 확인 전까지는 "복제 여부 불명"으로 취급하고 데이터 유실 가능성을 보수적으로 가정**.

### 4.3 격리 및 트래픽 전환 (Isolation / Failover)
- 장애 리전으로의 신규 트래픽 라우팅 중단 — **구체 메커니즘(DNS TTL, GSLB 등) 미확인**.
- DB/vault의 리더가 장애 리전에 있었다면 승격(promote) 절차 필요 — **현재 문서/코드에 승격 절차 없음. 이 부분이 가장 시급한 공백.**
- Kafka 프로듀서 재타겟팅 — 클러스터 토폴로지 확인 후 결정.

### 4.4 데이터 정합성 확인 (Consistency Check)
- **Charge 중복 여부 점검**: idempotency key가 강제되지 않으므로(`handlers/charge.go` FIXME), 페일오버 전후로 재시도된 요청이 중복 승인되지 않았는지 vault/원장 대사(reconciliation)로 확인 필요. 이는 이 코드베이스에서 리전 장애 대응 시 **가장 구체적이고 확실한 리스크**다.
- 정산 파이프라인(Kafka)에서 유실/중복 메시지 여부 확인.
- 환불 웹훅 재전송 여부 확인 — 단, 환불 흐름 자체가 §3에서 언급한 대로 미확정이므로 프로바이더 측 웹훅 재전송 정책도 별도 확인 필요.

### 4.5 커뮤니케이션
- 내부: 인시던트 채널, 결제/정산 영향 고객 공지 여부 — **에스컬레이션 정책/온콜 순번은 저장소에 없음, 별도 확인 필요.**
- PAN 미로깅 규칙(`CLAUDE.md`) 준수: 인시던트 중 임시 디버그 로깅을 늘릴 때도 PAN이 로그에 남지 않도록 재확인.

### 4.6 복구 및 재개
- 장애 리전 복구 후 트래픽 점진 복귀(canary) — **자동화 여부 불명**.
- 정산 cron이 아직 배선되지 않은 상태이므로, 장애로 밀린 정산 작업을 수동으로 트리거해야 할 가능성이 높음 (수동 재처리 절차 필요, 현재 없음).

### 4.7 사후 분석 (Postmortem)
- 중복 charge 발생 여부, 데이터 유실 여부, 복구 소요 시간(RTO) 및 데이터 손실 허용치(RPO) 대비 실제 값 기록.
- RTO/RPO 목표치 자체가 현재 문서화되어 있지 않음 — 이번 초안 작성을 계기로 목표치를 조직 차원에서 정의할 것을 권고.

## 5. 이 코드베이스 고유의 우선순위 리스크

1. **[높음] Idempotency key 미강제** (`handlers/charge.go` FIXME) — 리전 장애/페일오버 상황은 클라이언트 재시도가 몰리는 시점이라 중복 결제 리스크가 평시보다 커진다. 장애 대응 계획보다 선행해서 고쳐야 할 가능성이 있음.
2. **[높음] 정산 cron 미배선** (`main.go` TODO, 빈 `main()`) — 자동화된 정산 파이프라인 자체가 없으므로, 리전 장애 시 "정산을 잠시 멈췄다가 재개"할 스위치가 없다. 수동 개입이 전제된 상태.
3. **[중간] DB/Vault 페일오버 절차 부재** — 코드/문서상 근거 전무. 가장 먼저 인프라팀에 확인해야 할 항목.
4. **[중간] Kafka 클러스터 리전 토폴로지 불명** — 정산 데이터 유실/중복 가능성 판단의 전제.
5. **[낮음, 확인 필요] 환불 처리 방식 불확정(conflict 플래그)** — 장애 대응 절차의 환불 관련 스텝을 구체화하기 전 원 결정을 재확인해야 함.
