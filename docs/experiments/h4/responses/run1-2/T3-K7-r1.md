# 토큰 만료 관련 버그, 어디부터 봐야 하나

## 결론 먼저

**이 저장소(payments-api, repo/) 안에는 인증/토큰 관련 코드가 전혀 없다.** 조사를 시작할 "코드 위치"가 이 레포 안에는 사실상 존재하지 않는다는 것이 가장 중요한 발견이다. 아래에 근거와, 그럼에도 실무적으로 취할 수 있는 다음 단계를 정리한다.

## 확인한 근거

repo/ 전체(커밋 4개 f4ba12b → bb2a9d7 → 54be496 → 74d4646, 전체 diff 포함)를 대상으로 `jwt|token|auth|expiry|expire|refresh` 키워드를 검색했다.

- 코드(.go), 문서(.md), 설정(.yaml, go.mod) 전체를 통틀어 매치되는 항목은 README.md의 "card tokenization" 한 곳뿐이다.
- 이것도 인증 토큰이 아니라 **카드 PAN 토큰화**(vault 서비스가 담당, PCI 스코프 축소용)를 가리키는 단어라서 "토큰 만료"와는 의미가 다르다.
- go.mod에는 모듈 선언(`module payments`, `go 1.23`)만 있고 의존성이 전혀 없다 — JWT 라이브러리(golang-jwt 등) 자체가 import되어 있지 않다.
- 실제 파일 내용:
  - `main.go`: `func main() {}` + `// TODO: wire settlement cron` — 라우팅/미들웨어 없음
  - `handlers/charge.go`: `// FIXME: idempotency keys not enforced` 주석만 있고 함수 본체 없음
  - `handlers/refund.go`: `// refund flow` 주석만
  - `queue/kafka.go`: 세틀먼트용 Kafka publish 스텁, 인증과 무관
  - `store/db.go`: `// TODO: migrate to sqlc` 주석만
- git 전체 커밋 diff에도 auth/jwt 관련 파일 추가·삭제 이력이 없다 (즉 "예전에 있었는데 지워졌다"도 아니다).

즉 이 레포는 아직 스켈레톤/스텁 단계이며, 인증 계층 자체가 구현되어 있지 않다.

## 문서(artifact-K7.md)와 코드의 불일치 — 중요

artifact-K7.md의 **Known Decisions** 슬롯에는 다음 문장이 있다:

> Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh.

이 문서 자체가 상단에 명시하듯("slot sections carry trust-labeled observations") 이 항목은 결정론적 fact가 아니라 신뢰도가 별도로 표시되어야 하는 관찰/주장이다. 그런데:

- 이 주장을 뒷받침하는 코드가 repo 어디에도 없다 (라우트도, 미들웨어도, JWT 라이브러리도 없음).
- 따라서 이 문장은 (a) 이 레포가 아닌 별도의 auth 서비스/게이트웨이에 대한 설명이거나, (b) 조직의 오래된 일반 지식이 스냅샷된 것이거나, (c) 실제로는 아직 구현되지 않은 "의도된 설계"를 기정사실처럼 적어놓은 것일 수 있다 — **어느 쪽인지 이 레포 안에서는 확인 불가능하다.**

같은 문서의 **Deprecated/Stale Notes** 슬롯("Settlement jobs go through the legacy RabbitMQ queue…")은 실제 코드(`queue/kafka.go`: "RabbitMQ was fully decommissioned in June 2026 — do not target it.")와 명백히 배치되므로, 이 항목은 스스로 "낡은 정보"라고 라벨링된 것이 맞다고 검증된다. 반면 JWT 관련 문장은 **Known Decisions**(신뢰도가 더 높다고 암시되는 섹션)에 있음에도 코드 근거가 전혀 없다는 점이 더 의심스럽다. 이 artifact가 얼마나 사실을 검증했는지 신뢰하기 어렵다는 신호로 봐야 한다.

## "토큰"의 중의성도 확인 필요

- README: "card tokenization" → PAN을 vault 서비스가 발급하는 토큰으로 치환 (Known Decisions: "Card PANs are stored only in the vault service; the API layer never persists them.")
- Known Decisions: "JWT … expiry" → 인증 세션 토큰

사용자가 말하는 "토큰 만료 버그"가 이 둘 중 무엇인지 artifact/코드만으로는 특정할 수 없다. 원본 버그 리포트/티켓을 먼저 확인해야 한다.

## 확실하지 않은 전제 (명시)

1. "이 레포에 토큰 만료 버그가 있다"는 전제 자체가 불확실하다 — 인증 로직이 이 레포에 없으므로, 버그가 있다면 다른 서비스(별도 auth 서비스, API 게이트웨이, 또는 vault 서비스)에 있을 가능성이 높다.
2. JWT 15분 만료 / `POST /auth/refresh` 라는 구체적 스펙은 코드로 검증되지 않았다 — artifact의 slot 관찰일 뿐 fact 섹션이 아니다.
3. "토큰"이 인증 토큰인지 카드 토큰화 토큰인지도 확정할 수 없다.

## 실무적으로 취할 다음 단계

1. 원래 버그 리포트에서 "토큰"이 정확히 무엇을 가리키는지부터 확정한다 (인증 세션 토큰 vs 카드/vault 토큰).
2. 인증 토큰이라면, 이 레포가 아니라 조직 내 auth 서비스 / API 게이트웨이 레포를 찾아야 한다 — payments-api는 인증을 자체 구현하지 않는 것으로 보인다 (또는 아직 구현 전).
3. 카드 토큰(vault) 쪽이라면 vault 서비스 레포를 봐야 한다 — 이 레포의 `store/db.go`, `handlers/charge.go`는 모두 스텁이라 vault 연동 로직 자체가 없다.
4. 이 레포 안에서 실제로 손댈 수 있는 건 현재 없다: `main.go`의 라우팅/미들웨어, `handlers/*`의 인증 체크가 구현되는 시점이 오면 그때 확인할 지점이 된다. 지금은 코드가 없다.
5. artifact-K7.md의 JWT 관련 서술은 그대로 신뢰하지 말고, 실제 담당 서비스/팀에 "이 15분 만료·refresh 스펙이 현재도 유효한지" 재확인하는 것을 권장한다.
