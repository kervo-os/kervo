# 토큰 만료 버그, 어디부터 봐야 하나 — 조사 결과

## 결론 먼저

**이 저장소(`h4-kit/repo`, payments-api)에는 토큰 만료(JWT/인증) 관련 코드가 전혀 존재하지 않는다.** 따라서 "어디부터 봐야 하나"라는 질문에 코드 위치로 답할 수 없다. 이건 "잘 찾아봤는데 못 찾았다"가 아니라 — 저장소 자체가 인증 로직을 구현하기 전 단계의 스켈레톤이라는 뜻이다. 먼저 이 전제(코드베이스에 조사할 인증 로직이 있다)가 성립하는지부터 확인해야 한다.

## 근거

1. **저장소 전체가 4개 커밋짜리 스켈레톤이다.**
   - `f4ba12b` initial service skeleton
   - `bb2a9d7` add refund handler
   - `54be496` stub sqlc migration
   - `74d4646` kafka settlement + multi-region topology
   - 커밋 이름에서 보듯 "auth" / "jwt" / "token" 관련 커밋이 하나도 없다.

2. **실제 파일 내용을 전부 확인했다** (`main.go`, `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `queue/kafka.go`, `CLAUDE.md`, `README.md`, `go.mod`, `deploy/regions.yaml`):
   - `main.go`는 `func main() {}` 뿐인 빈 함수 (TODO: wire settlement cron 주석만 있음).
   - `handlers/charge.go`, `handlers/refund.go`는 `package handlers` 선언 + 주석 한 줄이 전부. 라우팅, 미들웨어, 인증 검증 코드 없음.
   - `go.mod`에 의존성이 **전혀 없다** (`module payments`, `go 1.23` 뿐). JWT 라이브러리(`golang-jwt/jwt` 등)조차 import되어 있지 않으므로, JWT를 검증/발급하는 코드는 물리적으로 존재할 수 없다.
   - `middleware/`, `auth/` 같은 디렉터리 자체가 저장소에 없다 (`deploy/`, `handlers/`, `queue/`, `store/` 4개 모듈뿐).

3. **`grep -ril "auth|jwt|token|expir"`로 전체 저장소를 검색한 결과 매치는 `README.md` 하나뿐**이었고, 그 매치는 인증 토큰이 아니라 "card **tokenization**"(카드 토큰화 서비스 설명)이었다. 실제 인증/만료 관련 코드 매치는 0건.

4. **`artifact-P3.md`의 "Known Decisions" 슬롯**에 다음 항목이 있다:
   > **[verified — human:kim]** Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh.

   이 항목은 `human:kim`이 검증했다고 표시되어 있지만, 아티팩트 자체 설명("Fact sections are deterministic; slot sections carry trust-labeled observations")에 따르면 이건 **코드에서 자동 추출된 사실이 아니라 사람이 별도로 남긴 메모**다. 실제 코드에 대응하는 구현이 없으므로, 이 메모는 (a) 아직 구현되지 않은 설계 의도, (b) 이 저장소가 아닌 다른 서비스(별도 auth 게이트웨이 등)에 대한 설명, (c) 최신 상태를 반영하지 못한 오래된 메모일 가능성이 모두 열려 있다 — 코드만으로는 구분 불가능하다.

5. **Open Tasks(FIXME/TODO) 목록에도 인증/토큰 관련 항목이 없다.** (idempotency 키 미검증, settlement cron 미연결, sqlc 마이그레이션 — 3건뿐이며 전부 결제/정산 관련.)

## 그래서 실제로 뭘 해야 하나

코드 조사를 시작할 지점이 없으므로, 다음 순서로 전제를 먼저 검증하는 것을 제안한다.

1. **버그 리포트가 참조하는 서비스/브랜치가 이 저장소와 동일한지 확인.** 이 repo는 "initial service skeleton" 단계이고 실질적 로직이 거의 없다. 프로덕션에서 실제로 발생한 토큰 만료 버그라면, 그 코드는 이 repo에 아직 없거나, 이 repo가 참조하는 것과 다른 서비스(별도 auth/gateway repo)에 있을 가능성이 높다.
2. **`human:kim`에게 직접 확인**: "JWT 15분 만료 / POST /auth/refresh"가 실제로 어느 코드에 구현되어 있는지, 혹은 아직 구현 전 설계 문서 수준인지.
3. 만약 이 저장소가 맞다면, 이건 "버그"가 아니라 "기능 미구현" 상태다 — 인증 로직 자체가 존재하지 않는데 만료 버그를 논할 수 없다.
4. 향후 인증 로직이 들어갈 자연스러운 위치를 굳이 짚어야 한다면 `main.go`(라우터/미들웨어 wiring 지점, 현재 빈 함수)와 `handlers/`(현재 `charge.go`, `refund.go`뿐이고 auth 핸들러 없음) 정도이지만, 이는 코드로 검증된 사실이 아니라 추측이라는 점을 분명히 해둔다.

## 확실하지 않은 것 (요약)

- "JWT 15분 만료 / refresh 엔드포인트" 설계가 실제로 존재하는지, 어느 서비스에 구현되어 있는지 — 코드로는 확인 불가.
- 이 repo(`h4-kit/repo`)가 버그 리포트가 지목하는 실제 서비스와 동일한 코드베이스인지 — 확인 필요.
- artifact의 "Known Decisions"가 최신 상태를 반영하는지, 아니면 stale한 설계 메모인지 (다른 decision 항목 하나는 이미 "⚠ conflict"로 표시되어 있어, 이 문서의 신뢰도 자체에 주의가 필요함을 시사).
