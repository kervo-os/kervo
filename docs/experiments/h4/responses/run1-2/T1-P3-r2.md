# 정산 잡 큐잉 개선 스케치 (payments-api)

## 0. 먼저 짚을 것 — 전제 확인

이 과제는 "기존 큐잉 코드를 개선"하는 게 아니라 **사실상 최초 구현**이다. 리포를 직접 열어본 결과:

- `main.go` = `func main() {}` (TODO 주석 하나뿐, cron 없음)
- `queue/kafka.go` = `func PublishSettlement() {}` (실제 Kafka 클라이언트 호출 전무, 토픽명만 주석)
- `store/db.go` = 주석 한 줄(`// TODO: migrate to sqlc`), 실제 스키마/쿼리 없음

즉 "큐잉 개선"이라는 표현과 달리 개선할 기존 로직 자체가 없다. 아래 스케치는 artifact-P3.md와 코드에서 확인한 사실을 근거로 한 **제안**이며, 스펙으로 확정된 것이 아니다.

### artifact와 코드가 충돌하는 부분 정리

- **Deprecated 섹션**: "정산은 레거시 RabbitMQ(`settlement_jobs`)를 통한다"는 stale로 표시돼 있고, 그 자체가 "superseded by Kafka 이전(2026-06)"이라고 명시. `queue/kafka.go`의 주석("RabbitMQ was fully decommissioned in June 2026 — do not target it.")과 README가 이를 뒷받침 → **현재 스택은 Kafka로 확정**, RabbitMQ 관련 코드는 작성하지 않음.
- **Known Risks의 "정산 cron 재시도 로직 없음"**: `[generated — backend:gpt-oss-120b]`, 즉 검증되지 않은 추정(근거는 단순히 open task 목록). 다만 코드를 직접 확인하니 main.go가 실제로 빈 스텁이라 이 추정과 결론은 일치. 그러나 "재시도 로직이 있었는데 빠졌다"가 아니라 "애초에 cron도, 재시도도, 아무것도 없다"가 정확한 현재 상태.
- **Known Decisions의 환불 관련 항목**(⚠ conflict 표시): 정산 큐잉과 직접 관련은 없어 보이지만 conflict 플래그가 있어 이번 스케치에서는 참고하지 않음. 환불이 정산 배치에 조정 항목으로 들어가는지는 불확실.
- **go.mod에 의존성이 전혀 없음**: Kafka 클라이언트 라이브러리가 아직 팀 차원에서 정해진 바 없다. 아래 코드는 `segmentio/kafka-go`를 가정(순수 Go, cgo 불필요, 널리 쓰임)했을 뿐 확정 사실이 아니다. `confluent-kafka-go`/`franz-go`를 쓴다면 API가 달라진다.
- **deploy/regions.yaml**: 리전 이름 3개(`us-east-1`, `eu-west-1`, `ap-northeast-2`)만 나열, active-active인지 active-passive인지, 오케스트레이션(K8s 등) 정보 전무. 멀티리전에서 cron 중복 실행을 어떻게 막을지는 문서화된 결정이 없어 아래에서 "제안"으로만 다룬다.

`handlers/charge.go`의 멱등키 FIXME는 정산 큐잉과 별개 이슈이지만, 상류(charge)에서 중복 요청이 걸러지지 않으면 정산 배치 대상에 중복 항목이 들어올 수 있다는 점에서 연관은 있다. 이번 스케치는 정산 잡 자체의 멱등키(merchant×기간×리전 기반)로 이를 별도 방어하되, charge.go FIXME 해결을 대체하지는 않는다.

## 2. 개선 방향

1. **Producer 실체화**: 빈 `PublishSettlement()`를 실제 Kafka writer 호출로 교체.
2. **멱등키**: 정산 잡 자체에 `merchant × region × period` 기반 결정적 키 부여 → 메시지 키로 사용(파티셔닝 + 중복 발행 방지 동시 해결).
3. **재시도 + 실패 격리**: writer 레벨 재시도 + 상위 레벨 백오프, 배치 중 개별 잡 실패가 전체 배치를 막지 않도록.
4. **cron 배선**: `main.go` TODO를 실제 스케줄러로 교체하고, 멀티리전 중복 실행 방지를 위한 리스(lease) 개념 도입(제안, 미확정).
5. **PAN 미노출**: CLAUDE.md 규칙("Never log PANs")에 따라 정산 잡/로그에는 merchant ID, 금액, 토큰화된 참조만 담고 카드 원본 데이터는 애초에 넣지 않음.
6. **테스트**: 팀 컨벤션(table-driven test)에 맞춘 테스트 스케치.

컨슈머 측 dedup/DLQ 소비 로직은 리포에 해당 서비스 코드가 아예 없어 이번 스케치 범위 밖으로 뒀다(별도 서비스일 가능성, 확인 필요).

## 3. 코드 스케치

### 3.1 `queue/kafka.go` — Producer 실체화

```go
package queue

import (
	"context"
	"crypto/sha256"
	"encoding/hex"
	"encoding/json"
	"fmt"
	"time"

	"github.com/segmentio/kafka-go" // 가정: 팀에서 확정한 클라이언트 아님. go.mod에 의존성 없음.
)

const (
	SettlementTopic    = "settlement.v1"     // 기존 주석에서 확인된 실제 토픽명
	SettlementDLQTopic = "settlement.v1.dlq" // 신규 제안 — 기존에 없음
)

// SettlementJob은 정산 배치 한 건에 대응하는 큐 페이로드다.
// CLAUDE.md 규칙("Never log PANs")에 따라 카드 원본 데이터는 절대 포함하지 않는다.
type SettlementJob struct {
	IdempotencyKey string    `json:"idempotency_key"`
	MerchantID     string    `json:"merchant_id"`
	Region         string    `json:"region"`
	PeriodStart    time.Time `json:"period_start"`
	PeriodEnd      time.Time `json:"period_end"`
	AmountMinor    int64     `json:"amount_minor"`
	Currency       string    `json:"currency"`
	EnqueuedAt     time.Time `json:"enqueued_at"`
}

// NewIdempotencyKey는 merchant×region×기간에 대해 결정적인 키를 만든다.
// cron이 같은 구간에 대해 재실행되거나 재시도되어도 같은 키가 나오므로
// 컨슈머 측(또는 브로커 압축)에서 중복 제거가 가능해진다.
func NewIdempotencyKey(merchantID, region string, periodStart, periodEnd time.Time) string {
	h := sha256.New()
	fmt.Fprintf(h, "%s|%s|%d|%d", merchantID, region, periodStart.Unix(), periodEnd.Unix())
	return hex.EncodeToString(h.Sum(nil))
}

type Producer struct {
	w *kafka.Writer
}

func NewProducer(brokers []string) *Producer {
	return &Producer{
		w: &kafka.Writer{
			Addr:         kafka.TCP(brokers...),
			Topic:        SettlementTopic,
			Balancer:     &kafka.Hash{}, // 같은 merchant/기간 → 같은 파티션, 병합 순서 보존
			RequiredAcks: kafka.RequireAll,
			MaxAttempts:  5, // 기존: 재시도 개념 자체가 없었음 (Known Risk에서 지적된 부분)
			BatchTimeout: 50 * time.Millisecond,
		},
	}
}

// PublishSettlement은 기존의 빈 스텁 `func PublishSettlement() {}`을 대체한다.
func (p *Producer) PublishSettlement(ctx context.Context, job SettlementJob) error {
	if job.IdempotencyKey == "" {
		return fmt.Errorf("settlement job missing idempotency key")
	}
	payload, err := json.Marshal(job)
	if err != nil {
		return fmt.Errorf("marshal settlement job: %w", err)
	}

	msg := kafka.Message{
		Key:   []byte(job.IdempotencyKey), // dedup 키 겸 파티션 키
		Value: payload,
		Headers: []kafka.Header{
			{Key: "region", Value: []byte(job.Region)},
		},
	}

	// kafka.Writer가 내부적으로 MaxAttempts만큼 재시도하지만,
	// 브로커가 그보다 오래 죽어있는 경우를 대비한 외부 백오프 루프.
	var lastErr error
	backoff := 100 * time.Millisecond
	for attempt := 0; attempt < 3; attempt++ {
		lastErr = p.w.WriteMessages(ctx, msg)
		if lastErr == nil {
			return nil
		}
		select {
		case <-ctx.Done():
			return ctx.Err()
		case <-time.After(backoff):
			backoff *= 2
		}
	}
	return fmt.Errorf("publish settlement job %s after retries: %w", job.IdempotencyKey, lastErr)
}

func (p *Producer) Close() error { return p.w.Close() }
```

### 3.2 `main.go` — cron 배선 (open task 해소)

```go
package main

import (
	"context"
	"log"
	"os"
	"os/signal"
	"syscall"
	"time"

	"payments/queue"
	"payments/store"

	"github.com/robfig/cron/v3" // 가정: cron 라이브러리도 팀 확정 사항 아님
)

func main() {
	ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
	defer stop()

	brokers := mustEnvList("KAFKA_BROKERS")
	producer := queue.NewProducer(brokers)
	defer producer.Close()

	db := store.MustOpen(os.Getenv("DATABASE_URL")) // store/db.go는 현재 스텁 — 아래 3.3 참고
	region := mustEnv("REGION")                      // us-east-1 | eu-west-1 | ap-northeast-2

	// 멀티리전 배포에서 cron을 모든 리전에서 동시에 돌리면 정산이 중복 발행된다.
	// DB 리스(lease)로 한 번에 한 리전만 활성 스케줄러가 되도록 하는 안(제안, 미확정 —
	// deploy/regions.yaml에는 리전 목록만 있고 active-active/passive 여부, 오케스트레이션
	// 방식에 대한 언급이 없음. k8s CronJob + 단일 활성 리전 지정 같은 대안도 가능).
	lease := store.NewSchedulerLease(db, "settlement-cron")

	c := cron.New()
	_, err := c.AddFunc("@every 1h", func() {
		runCtx, cancel := context.WithTimeout(ctx, 10*time.Minute)
		defer cancel()

		if !lease.TryAcquire(runCtx, region, 15*time.Minute) {
			log.Printf("settlement cron: lease held by another region, skipping run")
			return
		}
		defer lease.Release(runCtx)

		if err := runSettlementBatch(runCtx, db, producer, region); err != nil {
			// PAN은 절대 로그에 남기지 않는다 (CLAUDE.md) — merchant/job ID, 금액만 노출.
			log.Printf("settlement cron run failed: %v", err)
		}
	})
	if err != nil {
		log.Fatalf("wire settlement cron: %v", err)
	}
	c.Start()
	defer c.Stop()

	<-ctx.Done()
}

func runSettlementBatch(ctx context.Context, db *store.DB, p *queue.Producer, region string) error {
	pending, err := db.PendingSettlements(ctx, region) // store/db.go에 실제 구현 없음, 제안
	if err != nil {
		return err
	}
	for _, s := range pending {
		job := queue.SettlementJob{
			IdempotencyKey: queue.NewIdempotencyKey(s.MerchantID, region, s.PeriodStart, s.PeriodEnd),
			MerchantID:     s.MerchantID,
			Region:         region,
			PeriodStart:    s.PeriodStart,
			PeriodEnd:      s.PeriodEnd,
			AmountMinor:    s.AmountMinor,
			Currency:       s.Currency,
			EnqueuedAt:     time.Now().UTC(),
		}
		if err := p.PublishSettlement(ctx, job); err != nil {
			// 한 건 실패가 전체 배치를 막지 않음 — 다음 cron 사이클에 재시도됨
			// (같은 idempotency key라 중복 발행 걱정 없음).
			log.Printf("settlement job %s publish failed, will retry next run: %v", job.IdempotencyKey, err)
			continue
		}
		if err := db.MarkDispatched(ctx, s.ID, job.IdempotencyKey); err != nil {
			log.Printf("mark dispatched failed for %s: %v", s.ID, err)
		}
	}
	return nil
}
```

## 4. 이 스케치가 확정 짓지 않는 것 (팀 확인 필요)

- **Kafka 클라이언트 라이브러리**: `segmentio/kafka-go`로 가정했을 뿐, go.mod에 근거가 없다. `franz-go`/`confluent-kafka-go`(cgo, librdkafka 기반) 등 팀 표준이 있다면 API가 달라진다.
- **cron 실행 방식**: in-process 라이브러리(`robfig/cron`) + DB 리스로 스케치했지만, k8s CronJob을 리전 하나에만 배포하는 방식이 더 단순할 수 있다. `deploy/regions.yaml`에는 오케스트레이션 정보가 없어 판단 불가.
- **정산 단위**: "merchant × 기간(period)"으로 가정했다. README의 "settlement batching"이라는 표현과 도메인 관례에 기반한 추정이며, 코드/문서에 명시적 정의는 없다.
- **store 스키마**: `settlements` 테이블, `dispatched_at`/`idempotency_key` 컬럼 등은 전부 제안이다. sqlc 마이그레이션이 아직 안 끝나 실제 스키마 근거가 없다.
- **환불(refund) 관련 항목**: artifact에 conflict로 표시된 "환불은 provider webhook으로 비동기 처리" 결정이 정산 배치에 조정 항목으로 들어가는지 여부는 확인하지 못했다.
- **컨슈머 측 처리**: 이 리포에는 정산 잡을 소비하는 코드가 전혀 없다(프로듀서/스텁만 존재). DLQ 소비, 컨슈머 측 dedup 등은 별도 서비스일 가능성이 있어 범위 밖으로 뒀다.
