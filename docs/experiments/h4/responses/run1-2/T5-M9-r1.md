# PAN 로깅 관련 감사 체크리스트 — payments-api

## 0. 근거와 한계 (먼저 읽어야 함)

- 근거 자료: `h4-kit/artifact-M9.md` (kervo 컨텍스트 아티팩트) + `h4-kit/repo` 전체 파일 열람 + `git log -p`로 4개 커밋 전체 diff 확인.
- repo 실태: 커밋 4개(`f4ba12b` initial skeleton → `bb2a9d7` add refund handler → `54be496` stub sqlc migration → `74d4646` kafka settlement + multi-region topology). `.go` 파일은 전부 `package` 선언 + 주석 1~2줄뿐이며, 실제 핸들러 로직/로깅 호출이 전혀 없음(`grep -rniE "log|print"` 결과 코드 내 매치 0건, `go.mod`엔 모듈명과 go 버전만 존재하고 로깅 라이브러리 의존성 없음). CI/워크플로/린트/pre-commit 설정 파일도 repo에 없음.
- **따라서 이 체크리스트는 "지금 코드에서 위반을 발견했다"는 감사 결과가 아니라, 구현이 들어올 때 반드시 걸어야 하는 가드레일 체크리스트다.** 각 항목에 현재 확인 가능/불가능 여부를 표시했다.

## 1. 감사 착수 전 경고 — 아티팩트 신뢰도 문제

`artifact-M9.md`의 "Known Decisions" 슬롯 5개 중 **2개가 이미 코드로 반증**된 상태다.

- "Settlement jobs go through the legacy RabbitMQ queue" → 최신 커밋(`74d4646`, 2026-07-04)의 `queue/kafka.go`는 "RabbitMQ was fully decommissioned in June 2026 — do not target it"라고 명시하고 Kafka topic `settlement.v1`을 사용한다. **스테일 확정.**
- "Payments runs single-region in us-east-1; DR is cold-standby only" → 같은 커밋의 `deploy/regions.yaml`은 "active deployment topology (July 2026)"로 us-east-1/eu-west-1/ap-northeast-2 3개 리전을 명시한다. **스테일 확정.**

같은 슬롯에 있는 세 번째 문장 — **"Card PANs are stored only in the vault service; the API layer never persists them"** — 은 이 두 항목과 동일한 출처(캐시된 "Known Decisions")에서 나온 주장이다. 이미 그 중 40%가 반증된 이상, 이 문장도 **코드로 독립 검증되지 않은 미확인 전제로 취급해야 한다.** repo 어디에도 vault 클라이언트 호출, 토큰화 로직, DB 스키마가 존재하지 않아 이 주장을 코드로 확인할 방법이 없다. 이 문장을 사실로 전제하고 감사를 마쳤다고 결론 내리면 안 된다.

`CLAUDE.md`의 "Never log PANs"는 팀 규칙(정책) 선언일 뿐이며, 이를 강제하는 린터/pre-commit hook/CI 게이트/로그 스크러빙 미들웨어의 증거는 repo에 없다.

## 2. 체크리스트

### A. 구현 착수 전 설계 게이트
- [ ] charge 핸들러(`handlers/charge.go`, 현재 2줄 스텁)에 요청/응답 구조체를 정의할 때 PAN 필드에 마스킹/redaction(예: `MarshalLogObject`, 커스텀 `String()`, `json:"-"`)이 기본 적용되는가.
- [ ] 구조화 로깅 라이브러리 도입 시 PAN/CVV 필드 자동 redact 정책을 표준화하는가 (현재 `go.mod`에 로깅 라이브러리 자체가 없음 — 선택 단계에서 결정).
- [ ] "Never log PANs" 규칙을 코드 레벨에서 강제하는 정적 분석/커스텀 린터/pre-commit 계획이 있는가 (현재 부재).
- [ ] idempotency key 설계(`charge.go`의 FIXME: 미enforced) 시 키 소재로 raw PAN이나 그 단순 해시를 쓰지 않고, 키 값이 access log/트레이싱에 노출되지 않는가.

### B. 요청 수집(Ingress) 경로
- [ ] access-log 미들웨어가 요청 body/헤더 전체를 남길 경우 charge/refund 엔드포인트를 마스킹 또는 로깅 제외 대상으로 지정하는가.
- [ ] 입력 유효성 검사 실패 메시지가 원본 PAN 값을 그대로 echo하지 않는가.
- [ ] 리버스 프록시/게이트웨이 access log의 body 포함 여부 — **이 repo 범위 밖, 확인 불가.**

### C. 애플리케이션 로깅/에러 처리
- [ ] panic/recover 미들웨어가 스택트레이스와 함께 요청 body 전체를 로깅하지 않는가 (현재 recover 미들웨어 자체 없음).
- [ ] 디버그/verbose 로그의 "raw request dump" 코드가 프로덕션에서 비활성화되는가.
- [ ] 로그에 남기는 카드 식별자는 토큰(vault reference) 또는 마스킹된 PAN(앞6/뒤4)만 허용하는 필드 스키마가 문서화되어 있는가.

### D. 비동기 메시징 경로 (Kafka)
- [ ] `settlement.v1` 토픽 메시지 payload에 원본 PAN이 아닌 토큰화된 참조값만 포함되는가 — `PublishSettlement()`가 현재 빈 스텁이라 payload 스키마 자체가 미정, **스키마 확정 시 최우선 검토 대상.**
