# PAN 로깅 감사 체크리스트 — payments-api

## 0. 근거 자료 및 조사 범위

- 참조 문서: `h4-kit/artifact-P3.md` (kervo 자동 생성 컨텍스트 아티팩트)
- 참조 코드: `h4-kit/repo/` 전체 (`CLAUDE.md`, `README.md`, `main.go`, `handlers/charge.go`, `handlers/refund.go`, `queue/kafka.go`, `store/db.go`, `deploy/regions.yaml`, `go.mod`, `.gitignore`)
- 코드 전체를 직접 읽고 확인했으며, 아래 체크리스트는 이 저장소의 **실제 파일 내용**과 아티팩트의 **신뢰도 태그(verified/generated/observed/stale/conflict)**를 구분해 반영했다.

## 1. 가장 중요한 선행 사실: 이 저장소에는 아직 감사할 "로깅 코드"가 없다

실제로 파일을 열어본 결과, 이 저장소는 전부 스켈레톤/스텁 상태다.

- `handlers/charge.go` — `package handlers` + FIXME 주석 한 줄뿐, 함수 본문 없음
- `handlers/refund.go` — `package handlers` + 주석 한 줄뿐, 함수 본문 없음
- `queue/kafka.go` — `PublishSettlement()` 함수는 존재하나 본문이 비어 있음(`{}`)
- `store/db.go` — `package store` + TODO 주석뿐, DB 접근 코드 없음
- `main.go` — `func main() {}` 뿐, 로거 초기화/미들웨어 없음
- `go.mod`에 로깅 라이브러리(zap, logrus, slog 등) 의존성 없음

**따라서 "지금 코드에 PAN을 로깅하는 구문이 있는가"라는 질문에는 명확히 답할 수 있다 — 없다. 로그를 남기는 코드 자체가 아직 구현되지 않았다.** 이 체크리스트는 (a) 향후 구현 시 지켜야 할 게이트, (b) 지금 당장 확인/합의해야 할 정책·설계 전제로 구성된다. "현재 위반이 발견됨" 식의 결과는 아니므로 오해 없기를 바란다.

## 2. 신뢰도가 서로 다른 전제들 — 감사 전에 구분 필요

아티팩트의 슬롯 섹션은 트러스트 라벨이 다르므로, 이를 동일한 무게로 취급하면 안 된다.

| 내용 | 라벨 | 감사 시 취급 |
|---|---|---|
| "Card PANs are stored only in the vault service; the API layer never persists them." | **verified — human:kim** | 신뢰 가능한 저장(storage) 설계 원칙. 단, **로깅(logging)은 저장(persistence)과 별개 경로**이므로 이 결정이 "로그에도 PAN이 없다"를 보장하지 않음 — 별도 확인 필요 (근거: `CLAUDE.md`의 "Never log PANs." 규칙이 별도로 존재한다는 것 자체가, 저장과 로깅이 별도로 관리돼야 하는 리스크라는 뜻) |
| "Refunds are processed asynchronously via provider webhooks, not inline." | **verified — human:kim, ⚠ conflict 표시됨** | conflict 플래그가 있으므로 **그대로 신뢰하지 말 것**. 실제 refund 흐름(동기/비동기, 웹훅 페이로드 구성)은 `handlers/refund.go` 구현 시점에 재확인 필요 |
| "Settlement cron appears to lack retry logic" | **generated — backend:gpt-oss-120b** | AI 추론이며 human 검증 없음. `main.go`의 TODO(`wire settlement cron`)에서 나온 추정일 뿐, 재시도 로직 부재나 그로 인한 PAN 재로깅 위험을 단정할 근거는 아직 없음 |
| "Team convention: table-driven tests for all handler logic." | **observed — human:lee** | verified보다 낮은 신뢰도. 테스트 관례이므로 "테스트 픽스처에 실제/유사 PAN을 쓰지 않는다"는 것과는 별개로 확인 필요 |
| RabbitMQ(`settlement_jobs`) 관련 안내 | **stale — Kafka로 대체됨(2026-06)** | 무시. 신규 컨슈머는 Kafka `settlement.v1` 토픽 기준으로만 감사 |

**확실하지 않은 점**: refund 흐름에 conflict 플래그가 붙은 이유(무엇과 충돌하는지)는 아티팩트에 명시돼 있지 않다. 코드에도 refund 로직이 없어 확인 불가 — 담당자에게 직접 질의 필요.

## 3. PAN 로깅 감사 체크리스트

### A. 정책/설계 원칙 확인 (지금 바로 가능)

- [ ] `CLAUDE.md`의 "Never log PANs." 규칙이 팀 전체(신규 투입 인력 포함)에 온보딩 문서로 실제 전달되는지 확인
- [ ] "PAN"의 범위 정의 문서화 여부 확인 — PAN 전체, BIN(앞 6자리), 마지막 4자리(last4), 트랙2 데이터, CVV까지 포함하는지 팀 합의 필요 (현재 문서에 정의 없음 — **확인 필요**)
- [ ] 벌트(vault) 서비스가 PAN을 저장하는 유일한 지점이라는 결정(verified)이, "API 레이어는 PAN을 프로세스 메모리에서만 다루고 즉시 벌트로 넘긴다"는 데이터 흐름으로 구체화돼 있는지 확인 — 저장 안 한다는 것과 로그에 안 남긴다는 것은 다른 보장이므로 별도 설계 문서 필요

### B. HTTP 핸들러 계층 — `handlers/charge.go`, `handlers/refund.go` (구현 시 게이트)

- [ ] 요청/응답 본문 전체를 덤프하는 로깅 미들웨어를 두지 않는다 (access log에 body 포함 금지, 또는 PAN 필드 마스킹 필터 필수)
- [ ] `charge.go` 구현 시 FIXME(멱등키 미적용)를 해소하는 과정에서, 멱등키를 PAN이나 카드 원자재로부터 파생시키지 않는다 — 파생 시 로그에 찍히는 idempotency key 자체가 PAN 유추 가능한 값이 되지 않도록 확인
- [ ] 유효성 검증 실패/에러 응답 로그에 원본 요청 페이로드(PAN 포함 가능)를 그대로 포함하지 않는다
- [ ] 구조화 로깅(structured logging) 필드 화이트리스트 방식 채택 여부 확인 — 블랙리스트(민감 필드만 제외) 방식이 아니라 화이트리스트(허용 필드만 로깅) 방식을 권고
- [ ] `refund.go` 구현 시, PSP(결제대행사) 웹훅으로 수신한 raw payload를 디버그 로그에 그대로 남기지 않는다 (웹훅 payload에 PAN/트랙 데이터가 포함될 수 있음 — PSP 스펙 확인 필요)

### C. 메시지 큐 — `queue/kafka.go`

- [ ] `PublishSettlement()` 구현 시, Kafka 메시지 페이로드에 토큰(vault 참조값)만 포함하고 PAN 원문은 절대 포함하지 않는지 확인
- [ ] Kafka 프로듀서/컨슈머 라이브러리의 기본 로깅 레벨(debug 시 메시지 본문을 로그로 찍는 경우가 있음)이 프로덕션에서 비활성화돼 있는지 확인
- [ ] Dead-letter queue(DLQ) 처리 시 실패 메시지 전체를 로그/모니터링 시스템에 그대로 적재하지 않는지 확인 — 정산 실패 디버깅 때 가장 흔한 PAN 유출 경로
- [ ] 멀티 리전 배포(`deploy/regions.yaml`: us-east-1, eu-west-1, ap-northeast-2) 구조상 Kafka 토픽이 리전 간 복제되는지, 복제 시 메시지 로그가 중앙 집중형 로그 저장소로 이동하며 규제 스코프(PCI DSS, 데이터 국외 이전)를 확대하지 않는지 확인

## 4. 확실하지 않은 것 (명시적 한계)

- 이 저장소는 스텁 상태라 **실제 로깅 위반 사례는 발견되지 않았다** — "위반 없음"이 아니라 "검사할 대상 코드가 아직 없다"는 뜻이다. 이 체크리스트는 예방적 게이트다.
- refund 흐름의 conflict 플래그가 무엇을 가리키는지 원문에 없어 확인 불가 — 팀에 직접 질의 필요.
- 국내(한국) 금융 규제 항목(F 섹션)은 법률/컴플라이언스 전문가 확인이 필요하며, 이 감사(코드 기반)만으로 결론을 내릴 수 없다.
- "PAN 정의 범위"(BIN, last4, 트랙2, CVV 포함 여부)에 대한 팀 합의 문서를 찾지 못했다 — CLAUDE.md의 "Never log PANs."가 이 모든 것을 포함하는지는 추정이다.
- go.mod에 웹 프레임워크/로깅 라이브러리 의존성이 전혀 없어, 향후 어떤 라이브러리를 채택하느냐에 따라 이 체크리스트의 구체적 구현 방법(미들웨어 훅 위치 등)은 달라질 것이다.

2. `handlers/charge.go`, `handlers/refund.go` 구현 PR에는 반드시 위 B 섹션 체크리스트를 PR 템플릿에 링크
3. CI에 카드번호 패턴 정규식 스캐너 도입 (G 섹션) — 코드가 아직 적으므로 지금 도입하는 비용이 가장 낮음
4. refund 흐름 conflict 플래그의 실제 배경을 담당자(kim)에게 확인
5. 국내 리전 관련 컴플라이언스 요건은 별도로 법무/컴플라이언스팀 검토 요청
