# 환불(Refund) 흐름 문서화 — payments-api

## 0. 결론 먼저

**현재 리포지토리에는 환불 흐름이 실질적으로 구현되어 있지 않다.** `handlers/refund.go`는 패키지 선언과 주석 한 줄이 전부인 스텁 파일이며, 요청/응답 스키마, 라우팅, provider 연동, DB/큐 연동 로직이 전혀 없다. 아래 문서는 (a) 코드에서 실제로 확인된 사실, (b) 컨텍스트 아티팩트(`artifact-P3.md`)에 인간이 기록했지만 코드로는 검증 불가능한 주장, (c) 불확실하여 확인이 필요한 항목을 명확히 구분해서 정리한 것이다.

## 2. 컨텍스트 아티팩트(artifact-P3.md)에서 나온 정보 — 코드로 검증 불가

`artifact-P3.md`의 "Known Decisions" 섹션에 다음 항목이 있다:

> **[verified — human:kim]** ⚠ conflict
> Refunds are processed asynchronously via provider webhooks, not inline.

- 이 항목은 사람(kim)이 "verified"로 표시했지만, 문서 자체에 **"⚠ conflict" 플래그**가 붙어 있다. 즉 시스템/작성자가 이 결정이 다른 근거와 상충한다고 표시해 둔 상태다. 무엇과 충돌하는지는 아티팩트에 명시되어 있지 않다.
- 현재 코드베이스(`handlers/refund.go`)에는 provider webhook 처리 로직도, 동기/비동기 처리 로직도 전혀 없으므로 **이 주장을 코드로 확인하거나 반박할 근거가 없다.**
- 따라서 이 내용은 "확정된 설계"가 아니라 "충돌 플래그가 붙은, 검증 대기 중인 인간 진술"로 취급해야 한다. 실제 구현 시 이 결정이 여전히 유효한지 팀에 재확인이 필요하다.

참고로 함께 기록된 다른 "verified" 결정들(환불과 간접적으로 관련될 수 있음, 단 conflict 플래그 없음):
- **PAN 저장**: 카드 PAN은 vault 서비스에만 저장되며 API 레이어는 절대 영속화하지 않는다. (CLAUDE.md의 "Never log PANs"와 일치하는 방향의 정책)
- **인증**: JWT, 15분 만료, `POST /auth/refresh`로 갱신. 환불 엔드포인트도 이 인증 체계를 따를 것으로 추정되나, 코드상 실제 미들웨어/라우트 정의가 없어 확인 불가.

또한 "Deprecated / Stale Notes"에는 "정산 작업이 레거시 RabbitMQ 큐(`settlement_jobs`)를 통한다"는 내용이 **stale(구식)** 으로 표시되어 있고, `queue/kafka.go`의 실제 주석과 대조해도 이 stale 판정은 코드로 뒷받침된다(Kafka로 이전, RabbitMQ 폐기 확인됨). 다만 이는 **정산** 흐름 얘기이며 환불 흐름과는 별개다.

## 3. 확실하지 않아 확인이 필요한 전제

1. **환불 트리거 방식**: 아티팩트의 conflict 플래그가 붙은 진술("비동기, provider webhook 기반") 외에는 환불이 동기 API 호출인지, webhook 기반인지, 아니면 별도 큐/크론을 타는지 코드 근거가 전혀 없다.
2. **Idempotency**: `charge.go`에는 idempotency 키 미적용 FIXME가 명시돼 있지만, `refund.go`에는 동일한 언급이 없다. 환불도 중복 실행 방지가 필요한 영역이지만(중복 환불은 금전적 리스크가 큼), 현재 코드/문서 어디에도 환불의 idempotency 처리 여부에 대한 언급이 없다. **이 부분은 추정이 아니라 "정보 없음"으로 남겨야 한다.**
3. **환불과 정산(Kafka)/settlement cron의 관계**: README와 `queue/kafka.go`, `main.go`의 TODO는 모두 "settlement"(정산 배치)에 대한 것이며, 환불이 이 파이프라인을 공유하는지, 별도 경로인지 확인할 근거가 없다.
4. **환불과 리전 토폴로지의 관계**: `deploy/regions.yaml`은 배포 리전만 나열할 뿐, 환불 처리가 리전별로 어떻게 분산/집중되는지는 알 수 없다.
5. **DB 스키마**: `store/db.go`가 스텁 상태(`sqlc` 마이그레이션 TODO)이므로 환불 기록을 어느 테이블/스키마에 남기는지 알 수 없다.

## 4. 권장 다음 단계 (참고용)

- `handlers/refund.go` 실제 구현 착수 전, **conflict 플래그가 붙은 "provider webhook 비동기" 결정**을 kim 등 관련자와 재확인하여 conflict를 해소할 것.
- `charge.go`의 idempotency FIXME가 환불에도 적용되는 리스크인지 명시적으로 논의하고, 필요 시 refund 쪽에도 동일한 이슈를 티켓화할 것.
- 환불이 정산(Kafka) 파이프라인과 연동되는지 여부를 설계 단계에서 명확히 하고, 연동된다면 `queue/kafka.go`에 해당 토픽/컨슈머를 문서화할 것.
- PAN 미저장/미로깅 정책(CLAUDE.md, vault 결정)은 환불 로직 구현 시에도 그대로 적용되어야 함 — 환불 사유/메타데이터 로깅 시 PAN이 섞여 들어가지 않도록 코드 리뷰 체크리스트에 포함 권장.

## 5. 요약

- **핵심 발견**: "환불 흐름"이라 부를 만한 코드가 아직 존재하지 않는다 — `handlers/refund.go`는 2줄짜리 스텁이며, 관련 커밋(`bb2a9d7`)도 그 이상을 추가하지 않았다.
- **유일하게 존재하는 설계 진술**("provider webhook 기반 비동기 처리")은 인간이 검증했다고 표시되어 있지만 문서 자체가 conflict로 플래그했고, 코드로 뒷받침되지 않는다.
- Kafka/정산 크론/idempotency FIXME 등은 모두 **정산(charge/settlement)** 관련 사실이며 환불과의 연관성은 현재 근거가 없으므로 추정하여 문서화하지 않았다.
