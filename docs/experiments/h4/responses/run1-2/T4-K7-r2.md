# 환불(Refund) 흐름 문서 — payments-api

## 0. 결론 먼저

**현재 저장소 코드에는 환불 흐름이 구현되어 있지 않다.** `handlers/refund.go`는 패키지 선언과 주석 한 줄(`// refund flow`)뿐이며, 실제 로직·라우팅·큐 발행·DB 접근 코드가 전혀 없다. 아래 문서는 (1) 코드에서 확인 가능한 사실, (2) 컨텍스트 아티팩트(K7)가 주장하지만 코드로는 검증되지 않는 전제, (3) 아티팩트와 코드가 서로 모순되는 지점을 구분해서 기록한다.

## 1. 코드 사실 기반 확인 (Verified — 직접 읽은 코드)

- `handlers/refund.go` (전체 내용):
  ```go
  package handlers
  // refund flow
  ```
  이게 전부다. 함수 시그니처, HTTP 핸들러 등록, provider 연동, DB 호출 등 아무 구현이 없다.

- Git 히스토리 확인(`git log`, `git show bb2a9d7`): 커밋 `bb2a9d7` "add refund handler"가 추가한 diff는 정확히 위 2줄뿐이다. 즉 이 파일은 처음부터 지금까지 스텁 상태이며, 이후 구현이 붙었다가 삭제된 흔적도 없다.

- `main.go`: `func main() {}` — 서버 부팅, 라우터 등록, 미들웨어 연결 코드가 없다. 따라서 `/refund` 같은 엔드포인트가 실제로 어디에 어떻게 매핑되는지는 코드상 확인할 수 없다(=매핑 자체가 아직 없다고 보는 것이 타당).

- `queue/kafka.go`: `PublishSettlement()` 함수만 존재(빈 함수). 환불 관련 큐 발행 함수(예: `PublishRefund`)는 없다. 주석: "Settlement jobs are produced to the Kafka topic `settlement.v1`. RabbitMQ was fully decommissioned in June 2026 — do not target it."

- `store/db.go`: `// TODO: migrate to sqlc` 주석만 있는 스텁. DB 스키마, 쿼리, 환불 레코드 저장 로직 없음.

- `go.mod`: `module payments`, `go 1.23` — 외부 의존성 0개. HTTP 프레임워크, DB 드라이버, Kafka 클라이언트 라이브러리가 전혀 import되어 있지 않다. 이는 "아직 실제 네트워크/영속성 코드가 하나도 붙지 않았다"는 정황을 강하게 뒷받침한다.

- `handlers/charge.go`: `// FIXME: idempotency keys not enforced` — 이 FIXME는 **charge(결제) 핸들러**에 달려 있고, refund 핸들러에 달린 것이 아니다. 다만 결제와 환불은 통상 같은 안전장치(idempotency)가 필요한 영역이므로, 향후 환불 로직을 구현할 때 동일한 문제(중복 환불 위험)가 재현될 가능성이 있다는 점은 참고할 만하다 — 이건 코드 사실이 아니라 유추임을 명시한다.

- `CLAUDE.md`: 팀 규칙 "Never log PANs." — 환불 처리 로직을 구현할 때도 카드번호(PAN)를 로그에 남기지 않아야 한다는 원칙이 적용되어야 한다.

## 2. 아티팩트(K7)의 주장 — 미검증(trust-labeled) 전제

아티팩트 문서 자체가 "slot 섹션은 신뢰-라벨이 붙은 관찰이며 fact 섹션과 달리 결정론적이지 않다"고 명시하고 있다. 아래 항목들은 **코드로 직접 확인되지 않았다.**

- **"Refunds are processed asynchronously via provider webhooks, not inline."** (Known Decisions 슬롯)
  → 이것이 이 문서에서 가장 중요한 미확인 전제다. 코드에는 webhook 리스너, provider 콜백 라우트, 비동기 처리(큐 컨슈머, 워커 등)에 해당하는 어떤 코드도 없다. 즉 이 문장을 "설계 의도/과거 결정"으로는 기록할 수 있지만, **현재 구현 상태를 나타내는 사실은 아니다.** 확실하지 않다.

- **"Card PANs are stored only in the vault service; the API layer never persists them."**
  → 환불 시 PAN 접근/마스킹 처리와 관련이 있을 수 있는 결정이지만, `store/db.go`가 스텁이라 API 레이어가 실제로 PAN을 만지는지 여부를 코드로 검증할 수 없다. 미확인.

- **"Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh."**
  → 환불 API가 인증을 요구할 것으로 추정되나, 인증 미들웨어나 라우트 코드가 저장소에 없어 확인 불가.

- **Doc Summaries 슬롯: "Team convention: table-driven tests for all handler logic."**
  → `handlers/refund.go`, `handlers/charge.go` 모두 테스트 파일이 없다(저장소 파일 목록에 `_test.go` 없음). 관행이 문서화되어 있으나 refund 핸들러에는 아직 적용된 흔적이 없다.

## 3. 아티팩트 ↔ 코드 모순 (반드시 짚어야 할 지점)

- 아티팩트의 **"Deprecated / Stale Notes"** 슬롯: *"Settlement jobs go through the legacy RabbitMQ queue (`settlement_jobs`); new consumers should bind there."*
  → 이는 실제 코드(`queue/kafka.go` 주석: *"RabbitMQ was fully decommissioned in June 2026 — do not target it."*)와 **정면으로 모순**된다. 아티팩트 스스로 이 항목을 "Stale"로 분류해 놓았으므로 정합성은 맞지만, 이 노트가 여전히 문서에 남아있어 혼동을 유발할 수 있다.
  → **환불 흐름이 향후 큐 연동(예: 환불 완료 이벤트 발행)을 필요로 한다면, RabbitMQ/`settlement_jobs`가 아니라 Kafka 토픽 `settlement.v1` 계열을 따라야 한다.** 다만 refund 전용 토픽이 별도로 존재하는지는 코드에 없어 확인 불가.

## 5. 정리: 지금 시점에 "환불 흐름"으로 문서화할 수 있는 것 vs 없는 것

| 항목 | 상태 |
|---|---|
| HTTP 엔드포인트(`/refund` 등) 존재 여부 | 코드에 없음 — 확인 불가 |
| 환불 요청 → 처리 로직 | 없음 (`handlers/refund.go`는 주석 1줄) |
| provider webhook 비동기 처리 | 아티팩트 주장(미검증), 코드 없음 |
| 환불 관련 큐 발행/컨슈머 | 없음 |
| 환불 레코드 DB 스키마/저장 | 없음 (`store/db.go` 스텁) |
| idempotency 보장 | charge.go에 FIXME 존재(환불 자체엔 명시 없음), 실질적으로 미보장으로 추정 |
| PAN 미로깅 원칙 적용 대상 | 팀 규칙(CLAUDE.md)상 적용되어야 하나 구현 코드 없어 준수 여부 확인 불가 |

**요약하면, 이 저장소의 "환불 흐름"은 커밋 메시지와 파일 이름(`add refund handler`, `refund.go`)만 존재하는 자리표시자(placeholder) 단계이며, 실제 동작 순서·상태 전이·외부 연동은 전혀 구현되어 있지 않다.** 향후 개발자가 구현할 때 확인/결정해야 할 것: (1) 아티팩트가 말하는 "비동기 provider webhook" 설계가 실제 요구사항인지 재확인, (2) 큐를 쓴다면 Kafka(`settlement.v1` 계열)로 타겟, RabbitMQ 금지, (3) charge와 동일한 idempotency 키 전략 적용, (4) PAN 비로깅 규칙 준수, (5) `store/db.go`의 sqlc 마이그레이션 완료 후 스키마 확정.
