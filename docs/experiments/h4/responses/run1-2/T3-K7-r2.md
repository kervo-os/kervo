# 토큰 만료 버그, 어디부터 봐야 하나

## "JWT 15분 만료 + POST /auth/refresh"는 어디서 나온 정보인가

이 정보는 코드가 아니라 `artifact-K7.md`의 **"Known Decisions" 슬롯**에서만 나온다:

> Auth is JWT with 15-minute expiry; refresh via POST /auth/refresh.

이 문서는 스스로 "Fact sections are deterministic; slot sections carry trust-labeled observations"라고 명시한다. 즉 "Known Decisions"는 fact가 아니라 **검증 수준이 낮은 관찰(observation)** 섹션이다. 그리고 이 신뢰도 우려는 추측이 아니라 같은 문서 안에서 실제로 반증된다:

- 같은 문서의 **"Deprecated / Stale Notes" 슬롯**은 "Settlement jobs go through the legacy RabbitMQ queue... new consumers should bind there"라고 적어 두었지만, 실제 코드(`queue/kafka.go`, 가장 최근 커밋 `74d4646`)는 정확히 반대로 "RabbitMQ was fully decommissioned in June 2026 — do not target it"라고 못박고 있다.

즉 이 문서의 slot 섹션 중 최소 1건이 이미 코드와 정면으로 모순되는 것으로 확인됐다. 같은 슬롯 계열인 JWT/만료 관련 서술도 동일하게 **미검증·구식일 가능성**을 배제할 수 없다.

## 그래서 어디부터 봐야 하나 (권장 순서)

1. **먼저 버그 리포트/재현 정보를 이 저장소가 아닌 다른 곳에서 찾아라.** 이 리포에 인증 코드가 없으므로, JWT 발급/검증/리프레시 로직은 별도의 인증 서비스나 API 게이트웨이(별도 저장소)에 있을 가능성이 높다. `README.md`/`CLAUDE.md`/`go.mod` 어디에도 그런 외부 서비스에 대한 언급이 없으므로, 팀에 "이 payments-api가 어떤 인증 서비스에 의존하는지" 확인이 필요하다.
2. **"Known Decisions" 슬롯 내용을 현재 코드 기준 사실로 취급하지 마라.** 이미 RabbitMQ 사례에서 slot 섹션이 코드보다 뒤처져 있음이 확인됐다. 15분 만료·리프레시 엔드포인트 서술이 여전히 유효한 설계 결정인지, 혹은 아직 구현되지 않은 계획/과거 논의의 잔재인지 작성자(또는 커밋 히스토리가 있는 실제 인증 서비스 저장소)에게 재확인하라.
3. **"버그"가 아니라 "미구현"일 가능성을 검토하라.** 커밋 히스토리상 이 서비스는 아직 "initial service skeleton" 단계이며 각 핸들러가 본문조차 없다. 인증 미들웨어가 아직 연결되지 않았다면, 이는 조사해서 고칠 "버그"가 아니라 애초에 존재하지 않는 기능일 수 있다.
4. **이 저장소 내에서 유일하게 시간/만료와 관련될 여지가 있는 것은 없다.** `main.go`의 TODO(정산 크론 연결)와 `handlers/charge.go`의 FIXME(멱등성 키)는 토큰 만료와 무관한 별개 이슈이므로 혼동하지 말 것.

## 확실하지 않은 전제 (명시)

- "토큰 만료 관련 버그"가 실제로 이 `payments-api` 저장소의 코드에서 발생한다는 전제는 **현재 코드로 뒷받침되지 않는다.**
- JWT 15분 만료 및 `POST /auth/refresh`가 실제 운영 중인 스펙인지, 아니면 문서(slot)에만 남아있는 과거/미래 계획인지는 이 저장소 범위에서 **검증 불가**하다.
- 인증 로직이 존재한다면 그것이 어느 저장소/서비스에 있는지는 이 컨텍스트 아티팩트와 코드만으로는 알 수 없다.
