# PAN 로깅 감사 체크리스트 — payments-api

## 0. 요약 및 사용 안내

이 체크리스트는 `h4-kit/artifact-P3.md`(컨텍스트 아티팩트)와 `h4-kit/repo`(payments-api 저장소) 코드를 근거로 작성했다. **가장 먼저 밝혀야 할 전제**: 현재 저장소는 실질적으로 스켈레톤 상태다. `handlers/charge.go`, `handlers/refund.go`, `store/db.go`, `main.go`, `queue/kafka.go` 모두 `package` 선언과 주석(TODO/FIXME) 한두 줄만 있고 실행 로직이 없다(`git log -p` 로 전체 히스토리를 확인해도 4개 커밋 전부 스텁 수준). 로깅 라이브러리도 `go.mod`에 선언되어 있지 않다(의존성 없음, `module payments`, `go 1.23`뿐).

따라서 이 문서는 두 가지 역할을 겸한다.
1. **지금 확인 가능한 것**: 정책·문서·의사결정 수준에서 PAN 로깅 리스크가 존재하는지.
2. **앞으로 구현이 들어올 때 게이트로 쓸 체크리스트**: 실제 핸들러/스토어/큐 코드가 채워지는 시점에 이 항목들을 다시 적용해야 한다. 지금은 대부분 "코드 미구현으로 확인 불가"로 표시했으며, 이를 "통과"로 오독해서는 안 된다.

불확실한 부분은 표에 명시했다. 특히 아티팩트의 `⚠ conflict` 표시가 붙은 결정 항목은 원문에 충돌 내용 자체가 드러나 있지 않아, 무엇과 충돌하는지 나는 알 수 없다 — 이 부분은 추측하지 않고 그대로 열린 질문으로 남긴다.

## 3. 미해결/추가 확인 필요 사항 (Open Questions)

1. **refund 처리 방식 충돌(⚠ conflict)**: 아티팩트가 "verified"로 표시하면서도 충돌 플래그를 남긴 이유가 무엇인지 원 소스에서 확인해야 한다. 웹훅 비동기 처리가 맞는지, 인라인 동기 처리 경로가 실제로 남아있는지에 따라 B4의 리스크 우선순위가 달라진다.
2. **vault 서비스 분리 원칙의 코드 검증 불가**: "API 레이어는 PAN을 절대 영속화하지 않는다"는 결정은 인간이 검증했다고 표시되어 있으나, 현재 저장소에는 이를 뒷받침할 연동 코드가 전혀 없다. 이 결정이 코드베이스 밖(다른 레포, 아키텍처 문서)에서 검증된 것인지, 아니면 아직 구현되지 않은 설계 의도인지 구분이 필요하다.
3. **로깅 프레임워크 미선정**: `go.mod`에 로깅 라이브러리가 없다는 것은, PAN 마스킹/필드 화이트리스트 같은 통제를 "설계 단계"에서부터 넣을 기회이자, 반대로 아직 아무 표준도 없다는 뜻이기도 하다. 프레임워크 선정 시점에 이 체크리스트의 B/C/D/G 항목을 설계 요건으로 먼저 반영할 것을 권고한다.
4. **인프라/파이프라인 범위**: F, D4 항목은 이 Go 저장소의 범위를 넘어선다(로그 집계, Kafka 브로커 설정 등은 별도 IaC/인프라 레포에 있을 가능성). 해당 레포를 별도로 확인해야 완전한 감사가 된다.

## 4. 우선순위 권고 (구현 착수 시점 기준)

1. **D1 (Kafka 정산 payload 스키마)** — `PublishSettlement`이 실제 구현되는 시점에 스키마 리뷰를 필수 게이트로 지정. 정산 배치는 카드 데이터와 가장 밀접한 흐름이다.
2. **B1/B2 (charge 핸들러 로깅 + idempotency FIXME 연계)** — idempotency 미강제 상태에서 디버깅 압력이 커지므로, idempotency 구현과 동시에 "임시 디버그 로깅 금지"를 코드리뷰 규칙으로 명문화.
3. **G2/G3 (공용 로거 wrapper + CI 시크릿 스캐너)** — 로깅 프레임워크 도입과 동시에 예방적 통제를 얹어야 사후 탐지에만 의존하지 않는다.
4. **E1 (vault 연동 로깅)** — vault 클라이언트 코드가 실제로 작성되는 첫 PR에서 반드시 검사.
5. **A3 (refund 충돌 플래그 규명)** — 코드와 무관하게 지금 바로 확인 가능한 항목이므로 우선 해소 권장.
