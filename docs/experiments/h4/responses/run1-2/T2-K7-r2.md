## 1. 아티팩트-코드 정합성 확인

`artifact-K7.md`는 "Deprecated / Stale Notes" 슬롯에 다음을 이미 스스로 낡은 정보로 표시해 두었습니다.

> 설정 정산 잡은 레거시 RabbitMQ 큐(`settlement_jobs`)를 통해 처리되며, 신규 컨슈머는 그쪽에 바인딩해야 한다.

코드를 보면 이 내용은 실제로 폐기된 경로입니다.

- `queue/kafka.go`: "Settlement jobs are produced to the Kafka topic `settlement.v1`. RabbitMQ was fully decommissioned in June 2026 — do not target it."
- `README.md`: "Settlement jobs flow through Kafka (queue/); deployed multi-region (deploy/regions.yaml)."
- 최근 커밋 `74d4646` (2026-07-04, "kafka settlement + multi-region topology")가 이 Kafka 전환과 멀티 리전 설정을 같은 커밋에서 도입했습니다.

즉 아티팩트 자체가 이 항목을 "Stale"로 올바르게 분류해 두었고, 코드도 이를 재확인해 줍니다 — **아티팩트와 코드가 상충하는 것이 아니라 일치합니다.** 이번 계획은 RabbitMQ를 대상으로 하지 않고, Kafka `settlement.v1` 토픽을 현재 정산 경로로 취급합니다. (다만 이 stale 항목이 아직 문서에 남아있는 이유는 불명 — 재생성/정리 후보로 별도 보고할 만합니다.)

아티팩트의 "Known Decisions"에는 리전 토폴로지나 큐 방식에 대한 주장이 없고, 다음 세 가지만 있습니다: (1) PAN은 vault 서비스에만 저장, API 레이어는 비저장, (2) JWT 인증 15분 만료 + `POST /auth/refresh`, (3) 환불은 provider webhook을 통한 비동기 처리. 이 세 항목은 저장소 코드로 직접 확인되지도, 반박되지도 않습니다(`handlers/refund.go`는 `// refund flow` 주석뿐인 스텁). 반박 근거가 없으므로 잠정 유효한 것으로 다루되, "코드로 검증됨"이 아니라 "문서 주장, 미검증" 상태로 아래 표에 표시합니다.

- 3개 리전이 active-active인지 active-passive(콜드/웜 스탠바이)인지, 그리고 어느 리전이 primary인지
- 리전 간 트래픽 라우팅/페일오버 메커니즘 (DNS 가중치, GSLB, Anycast, LB 헬스체크 등)

- vault 서비스 자체의 리전 이중화 여부
- 모니터링/알림/온콜 도구, 에스컬레이션 정책, 헬스체크 엔드포인트
- `deploy/regions.yaml`은 리전 "이름 목록"일 뿐 실제 배포 매니페스트(Terraform/K8s 등)가 저장소에 없어, 이 3개 리전이 실제로 무엇을 가리키는지(전체 스택 복제인지 일부 컴포넌트만인지)도 확정할 수 없습니다.

또한 Kafka 정산 경로와 멀티 리전 설정은 같은 커밋(`74d4646`, 2026-07-04, 어제)에서 함께 도입된 매우 최근 변경입니다. 장애 대응 이력이나 검증된 운영 실적이 없는 상태라는 점을 감안해야 합니다.

- **정산 재시도 로직 부재 추정** (artifact Known Risks) — 장애 후 재처리 전략을 별도 설계해야 합니다.

권고: 이 계획을 "운영 가능"으로 승인하기 전에 최소한 idempotency 강제는 해결하거나, 해결 전까지는 장애 시 자동 재시도를 금지하고 수동 검토를 의무화하는 임시 가드레일을 둘 것을 권고합니다.

## 4. 시나리오별 초기 대응 방향

### 4.1 단일 리전 완전 장애 (예: eu-west-1 다운)
- 해당 리전으로의 신규 트래픽 라우팅 차단이 필요하나, 실제 라우팅 계층이 미확인이므로 구체적 명령/콘솔 절차는 인프라팀 확인 후 채워야 합니다.
- Kafka: 장애 리전에 붙어 있던 프로듀서/컨슈머 상태 확인. 리전 간 복제 여부가 불명확하므로, 장애 리전에서 처리 중이던 정산 이벤트의 유실 가능성을 배제하지 말 것.
- 나머지 두 리전이 트래픽을 흡수할 용량(capacity)을 갖는지 사전 확인 필요 (오토스케일링/여유 용량 근거 없음).

### 4.2 Kafka(`settlement.v1`) 장애/지연
- 정산 배치가 지연·유실될 위험. `main.go`의 settlement cron이 아직 연결되지 않은 상태(TODO)이므로, 운영 환경에서 실제로 배포된 크론이 있는지부터 확인해야 합니다.
- 재처리 시 idempotency key가 강제되지 않으므로(`handlers/charge.go` FIXME), 큐 재처리·재시도 과정에서 **중복 정산·중복 청구 위험**이 큽니다. 재시도를 걸기 전 반드시 수동 중복 검증 절차를 넣을 것을 권고합니다.

### 4.3 DB/스토어 장애
- `store/db.go`가 스텁 상태라 실제 DB 엔진, 복제 구성, 백업/복구(RPO) 정보가 전혀 없습니다. 이 시나리오는 세부 절차를 채우기 전에 실제 DB 인프라 문서 확보가 선행되어야 합니다.

### 4.4 Vault 서비스(카드 PAN 조회) 리전 장애
- vault가 별도 서비스로 분리되어 있다는 것만 (아티팩트 근거로) 확인되고, 이중화 여부는 불명입니다. Vault 접근 불가 시 결제/환불 흐름이 어떻게 저하되는지 별도 확인이 필요합니다. `CLAUDE.md`의 팀 규칙("Never log PANs")은 장애 대응 중 임시 로깅/디버깅 시에도 반드시 지켜야 합니다.

### 4.5 환불(webhook) 경로 장애
- 환불이 provider webhook을 통한 비동기 처리라는 것은 아티팩트 주장이며 `handlers/refund.go`는 빈 스텁이라 실제 구현으로 확인되지 않습니다. 리전 장애 시 provider webhook의 목적지 라우팅(어느 리전 엔드포인트로 오는지)이 어떻게 처리되는지도 미확인 — 웹훅 유실/중복 재전송 정책을 provider 문서와 함께 별도 확인 필요.

## 5. 대응 절차 초안 (러너북 스켈레톤 — 다수 단계는 세부 확정 필요)

1. **감지**: 모니터링/알림 출처 미확인 — 확인 후 채움.
2. **트리아지/에스컬레이션**: 온콜 정책 미확인 — 확인 후 채움. 영향 리전, 영향 컴포넌트(결제/정산/환불) 최초 분류.
3. **트래픽 격리**: 장애 리전을 라우팅에서 제외. 구체 메커니즘은 4.1 참고, 확인 필요.
4. **정산 파이프라인 안전 정지/격리**: Kafka 컨슈머를 무조건 재가동하기 전에, idempotency 미강제 상태를 감안해 중복 처리 여부를 수동 확인.
5. **결제 경로 임시 가드레일**: idempotency 미강제 상태이므로, 페일오버 도중 클라이언트/게이트웨이 재시도를 제한하거나 수동 검토를 거치도록 임시 조치.
6. **데이터 정합성 점검**: 장애 리전에서 처리 중이던 트랜잭션의 유실/중복 여부 확인. PAN 미노출(vault 경계 유지) 재확인.
7. **복구/페일백**: 리전 복구 후 트래픽 점진 복귀, 정산 큐 재동기화 확인.
8. **사후보고(post-mortem)**: 근본 원인, 재발 방지 항목, 이 문서의 TBD 항목 업데이트. 관련 핸들러 테스트는 팀 컨벤션(table-driven tests, artifact Doc Summaries)에 맞춰 회귀 테스트로 추가.

## 8. 요약

- 이 저장소 코드 기준 현재 토폴로지는 **3개 리전(us-east-1, eu-west-1, ap-northeast-2) 멀티 리전 + Kafka(`settlement.v1`) 정산 파이프라인**이며, 아티팩트도 이 사실과 상충하지 않습니다(레거시 RabbitMQ 언급은 아티팩트 스스로 Stale로 표시한 항목이고 코드도 이를 확인해 줍니다).
