# payments-api 환불(Refund) 흐름 문서

## 0. 결론 먼저 (TL;DR)

**저장소 코드에는 환불 흐름이 실질적으로 구현되어 있지 않다.** `handlers/refund.go`는 패키지 선언과 주석 한 줄뿐인 스텁이며, 라우팅·HTTP 핸들러 함수·DB 접근·큐 연동·프로바이더(웹훅) 연동 코드가 전혀 존재하지 않는다. 아래 문서는 (a) 코드로 직접 확인된 사실과 (b) 컨텍스트 아티팩트(`artifact-K7.md`)가 주장하는 미확인 정보를 명확히 구분해서 정리한 것이다. "환불 흐름"이라고 부를 만한 실제 로직은 코드베이스에 없으므로, 이 문서는 **"현재 상태 진단 + 아티팩트상의 설계 의도"** 보고서에 가깝다.

## 2. 아티팩트(`artifact-K7.md`)가 주장하는 내용 — **코드로 미확인**

`artifact-K7.md`의 "Known Decisions" 슬롯(문서 자체 설명상 "trust-labeled observations", 즉 사실 섹션과 달리 검증 신뢰도가 다른 슬롯)에 다음 문장이 있다:

> Refunds are processed asynchronously via provider webhooks, not inline.

- 이 문장이 사실이라면 예상되는 아키텍처는: (1) 환불 요청 접수 → (2) 결제 프로바이더(PG사)에 환불 요청 전달 → (3) 프로바이더가 처리 완료 후 웹훅으로 콜백 → (4) 웹훅 수신 핸들러가 내부 상태를 업데이트. **그러나 이 4단계 중 어느 것도 코드에 존재하지 않는다.** 웹훅 수신 엔드포인트, 프로바이더 클라이언트, 상태 머신, 재시도 로직 모두 부재.
- 이 결정 사항은 "누가, 언제 확정했는지"에 대한 근거(ADR, PR, 코드 주석)가 저장소 내에 전혀 없다. **아티팩트가 어디서 이 사실을 얻었는지 추적 불가능** — 신뢰도를 확인할 방법이 없으므로 "확정된 설계"가 아니라 "구두 전달/외부 문서 기반의 미검증 진술"로 취급해야 한다.

## 3. 발견된 불일치 및 리스크

1. **커밋 메시지와 실제 diff의 불일치**: "add refund handler"라는 커밋 메시지는 실제 핸들러 구현을 시사하지만, 실제로는 2줄 스텁만 추가됨. 신규 투입자가 커밋 로그만 보고 "환불 기능이 이미 있다"고 오판할 위험이 있다.
2. **라우팅 근거 전무**: `main.go`가 완전히 비어 있어, 환불 엔드포인트의 HTTP 메서드/경로/미들웨어(인증 등)를 코드로 특정할 수 없다. CLAUDE.md에 언급된 JWT 인증(`POST /auth/refresh` 등, 15분 만료)이 환불 엔드포인트에도 적용되는지 불명.
3. **아티팩트 내부의 큐 관련 상충 정보 (환불과는 직접 관련 없으나 저장소 신뢰도에 영향)**: `artifact-K7.md`의 "Deprecated / Stale Notes" 슬롯은 "Settlement jobs go through the legacy RabbitMQ queue (`settlement_jobs`); new consumers should bind there."라고 되어 있으나, 실제 코드(`queue/kafka.go`, 2026-07-04 커밋)는 "RabbitMQ was fully decommissioned in June 2026 — do not target it."라고 명시하며 Kafka `settlement.v1` 토픽을 정본으로 지정한다. 이는 정산(settlement) 큐에 대한 내용으로 **환불 흐름 자체와는 무관**하지만, 같은 아티팩트 안에 최신 코드와 상충하는 stale 정보가 섞여 있다는 것은 "Known Decisions" 슬롯의 환불 관련 진술(§2)도 검증 없이 그대로 믿기 어렵다는 근거가 된다.
4. **idempotency 미보장(charge.go FIXME)이 환불에도 적용되는지 불명**: 결제 취소/환불은 중복 처리 시 금전적 피해가 직접적이므로 특히 중요한 리스크 포인트이나, 환불 쪽에 동일한 FIXME나 코드가 없어 확인 자체가 불가능하다.
5. **PAN 비저장 정책과 환불의 접점 불명**: "Card PANs are stored only in the vault service"(아티팩트 Known Decisions)라는 진술이 있으나, 환불 시 카드/토큰 참조를 어떻게 조회하는지 코드 근거 없음.

- 환불이 정산(Kafka `settlement.v1`) 파이프라인과 상호작용하는가, 아니면 완전히 별도 경로인가? 현재 코드상 연결 근거 없음.
