# 게이트웨이 인증 미들웨어 부착 시 전제할 토큰 정책 (만료·갱신)

## 0. 먼저 짚어야 할 전제 자체의 불확실성 (확신도: 낮음)

- 과제 문구는 "게이트웨이에 인증 미들웨어를 붙인다"고 전제하지만, 이 아티팩트에는 `gateway`라는 이름의 모듈/컴포넌트가 전혀 등장하지 않는다. Related Modules에 있는 것은 `deploy/`, `handlers/`, `queue/`, `store/`뿐이다. [근거: Fact — Related Modules]
  - 즉 "게이트웨이"가 (a) `handlers/`가 사실상 담당하는 진입점을 가리키는 것인지, (b) 문서에 기술되지 않은 별도 API 게이트웨이 서비스인지, (c) 멀티 리전 배포(`deploy/regions.yaml`) 앞단의 로드밸런서/엣지 계층을 가리키는 것인지 문서만으로는 구분할 수 없다. **이 전제 자체가 확인 필요 사항이며, 추정하지 않는다.**

## 1. 확정 가능한 토큰 정책 (확신도: 높음)

- 인증 방식은 JWT이며 만료(expiry)는 15분이다. [근거: Known Decisions — **verified, human:kim**, 충돌(conflict) 플래그 없음]
- 갱신(refresh)은 `POST /auth/refresh` 엔드포인트를 통해 이루어진다. [근거: Known Decisions — **verified, human:kim**, 충돌 플래그 없음]
- 이 두 진술은 사람(kim)이 검증한 것으로 라벨링되어 있고, 문서 내 다른 항목(환불 처리)에 붙은 것과 같은 ⚠ conflict 표시가 없다. 따라서 이 아티팩트 안에서는 가장 신뢰도가 높은 축에 속한다. 다만 "사람이 검증했다"는 라벨이 검증 절차·시점·검증자의 코드베이스 접근 범위까지 보증하지는 않으므로, 미들웨어 구현 전 코드(`handlers/`, 혹은 인증 관련 모듈)로 최종 대조가 필요하다.

미들웨어에 최소한으로 전제할 수 있는 것: **"Authorization 헤더의 JWT는 발급 후 15분 뒤 만료되며, 만료된 토큰으로의 재인증은 `POST /auth/refresh` 호출로 이루어진다"**는 문장 자체는 문서 근거상 신뢰할 수 있다.

## 2. 만료·갱신과 관련해 문서가 답하지 않는 것들 (확신도: 낮음 — 전부 불명)

아래는 게이트웨이 미들웨어를 실제로 구현하려면 반드시 필요하지만, 이 아티팩트에는 전혀 등장하지 않는 항목이다. 유추로 채우지 않는다.

- **리프레시 토큰 자체의 만료/수명**: `/auth/refresh`가 소비하는 리프레시 토큰이 별도로 존재하는지, 존재한다면 그 유효기간(예: 7일, 30일, 무기한)이 얼마인지 — 불명.
- **리프레시 토큰 회전(rotation) 여부**: 매 갱신마다 리프레시 토큰이 새로 발급되는 원-타임 토큰인지, 재사용 가능한 토큰인지 — 불명.
- **재사용 탐지/폐기(revocation)**: 탈취된 리프레시 토큰이 재사용될 경우 세션을 강제 폐기하는 로직이 있는지, 로그아웃 시 서버 측 블랙리스트/화이트리스트가 있는지 — 불명.
- **서명 알고리즘 및 키 관리**: HS256/RS256 등 알고리즘, 서명 키(secret/JWKS) 로테이션 정책 — 불명.
- **최초 발급(로그인) 엔드포인트**: 문서에는 갱신 엔드포인트(`POST /auth/refresh`)만 언급되어 있고, 최초 토큰 발급 경로(로그인 API)는 등장하지 않는다 — 불명.
- **토큰 전달/저장 방식**: Authorization 헤더 Bearer 토큰인지, 쿠키 기반인지(그렇다면 HttpOnly/Secure/SameSite 속성) — 불명.
- **클럭 스큐(clock skew) 허용치**: 15분이라는 짧은 만료 시간과 결합해, 검증 서버 간 시간 오차 허용 범위가 명시되어 있지 않음 — 불명.

- 가장 최근 커밋(`74d4646`, 2026-07-04)의 메시지가 "kafka settlement + multi-region topology"로, 멀티 리전 토폴로지가 최근에 도입/변경되었을 가능성을 시사한다. [근거: Fact — Recent Changes]

## 4. 인증 미들웨어가 처리해야 할 경로가 JWT 하나만은 아닐 가능성 (확신도: 중간)

- 환불(refund)은 프로바이더 웹훅을 통해 비동기로 처리된다는 결정이 문서에 있다. [근거: Known Decisions — **verified, human:kim**, 단 ⚠ conflict 플래그 있음 — 이 항목 자체의 최종 신뢰도는 불확실]
- 웹훅 콜백 경로는 통상 사용자 세션의 15분 JWT Bearer 토큰이 아니라 별도의 서명 검증(프로바이더 서명 시크릿 등)을 쓰는 경우가 많다. 다만 이 아티팩트는 그 방식을 명시하지 않는다.
- 따라서 "게이트웨이 인증 미들웨어 = 모든 인바운드 트래픽에 15분 JWT 정책 적용"이라고 일괄 전제하면 위험하다. **최소한 (a) 사용자/클라이언트발 API 트래픽과 (b) 프로바이더발 웹훅 콜백 트래픽을 구분해서 인증 전략을 설계해야 한다는 점**은 문서상 근거 있는 추정이지만(중간 확신도), 웹훅 인증의 구체적 메커니즘은 불명이다. 또한 이 추정의 전제인 환불 결정 항목 자체에 conflict 플래그가 있다는 점을 감안하면, 이 절 전체는 확정이 아니라 "구현 전 반드시 코드/담당자 확인 필요" 수준으로 취급해야 한다.

## 5. 간접적으로 참고할 만한 리스크 (확신도: 낮음, 토큰 정책과 직접 관련은 약함)

- `handlers/charge.go`에 idempotency key가 강제되지 않는다는 FIXME가 있다. [근거: Open Tasks — handlers/charge.go:2]
  - 만약 게이트웨이 미들웨어가 만료된 토큰에 대해 자동으로 `/auth/refresh` 후 원 요청을 재시도하는 방식으로 구현된다면, idempotency key 부재와 결합해 결제 요청이 중복 처리될 가능성이 있다. 다만 이는 문서에 직접 언급된 사실이 아니라 두 개의 별도 관찰(토큰 갱신 방식 + idempotency 미적용)을 결합한 추론이므로, **확신도를 낮게 표시하며 실제 코드 확인 전까지는 가설로만 취급**한다.
